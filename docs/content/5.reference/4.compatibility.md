## Compatibility Policy

Sockudo maintains two distinct compatibility contracts:

- **Protocol V1** is the Pusher-compatible contract.
- **Protocol V2** is the Sockudo-native contract.

This split is intentional. V1 exists to minimize migration risk for existing Pusher users. V2 exists to expose Sockudo-native features without forcing them into Pusher naming or semantics.

### Rules

1. **V1 must remain Pusher-compatible by default**
   - Use Pusher-compatible event prefixes, auth flows, webhook headers, and HTTP API semantics.
   - Existing Pusher clients and server SDKs should continue to work against Sockudo by changing host/configuration before requiring code changes.

2. **V2 must remain Sockudo-native by default**
   - Use `sockudo:` / `sockudo_internal:` prefixes and Sockudo-specific feature naming.
   - New V2-only features must be introduced under Sockudo-native naming, not Pusher naming.

3. **Do not mix semantics under one surface**
   - Pusher naming must not silently gain Sockudo-only behavior.
   - Sockudo naming must not pretend to be a Pusher compatibility layer unless it is explicitly documented as an alias.

4. **Sockudo-only features must be additive**
   - Features such as connection recovery, binary wire formats, delta compression, tag filtering, message extras, and extended auth flows should be opt-in or V2-only.
   - Compatibility-safe behavior should remain the default path for V1 integrations.

5. **Compatibility claims must be explicit**
   - `protocol-compatible` means Sockudo speaks the same wire/API/auth contract.
   - `source-compatible` means existing application code can be reused with minimal or no code changes.
   - `package/import-compatible` means the original dependency/module identity can remain unchanged.
   - Documentation and release notes should avoid claiming stronger compatibility than an SDK actually provides.

### SDK Guidance

- **Server SDKs**
  - V1-facing APIs should stay close to official Pusher server SDK semantics where practical.
  - V2-facing APIs may expose Sockudo-native methods, options, and payload extensions.
  - Sockudo-specific additions such as `idempotency_key` must be additive and must not break Pusher-compatible call patterns.

- **Client SDKs**
  - V1 behavior should feel like official Pusher clients.
  - V2 behavior should use Sockudo-native event prefixes, protocol fields, and feature flags.
  - V2-only capabilities should not silently alter V1 behavior.

### Migration Principle

The recommended migration path is:

1. keep existing Pusher-compatible clients and server SDKs on V1
2. switch them to Sockudo infrastructure
3. adopt Sockudo client/server SDKs where needed
4. opt into V2 only when Sockudo-native features are desired

In short:

- **V1 is for compatibility**
- **V2 is for Sockudo**

## Client Feature Compatibility

| Feature | pusher-js | Laravel Echo | `@sockudo/client` |
| --- | --- | --- | --- |
| Public/private/presence channels | Yes | Yes | Yes |
| Channel auth endpoint flow | Yes | Yes | Yes |
| User signin flow | Yes (manual wiring) | Partial | Yes |
| Client events (`client-*`) | Yes | Yes | Yes |
| Wire formats (`json`) | Yes | Yes | Yes |
| Wire formats (`messagepack`, `protobuf`) | No | No | Yes (V2) |
| Tag filter subscribe API | No | No | Yes (V2) |
| Automatic delta decode | No | No | Yes (V2) |
| Per-subscription delta options | No | No | Yes (V2) |
| Serial number tracking | No | No | Yes (V2) |
| `message_id` deduplication | No | No | Yes (V2) |
| Connection recovery (`sockudo:resume`) | No | No | Yes (V2) |
| Wildcard channel subscriptions | No | No | Yes (V2) |
| Metachannels (`[meta]channel`) | No | No | Yes (V2) |

## Server SDK Compatibility

| Capability | Sockudo server SDKs | Pusher server SDKs |
| --- | --- | --- |
| Event trigger | Native | Works (set Sockudo `host`/`port`) |
| Channel auth generation | Native | Works (`authorizeChannel`) |
| Webhook signature model | Same HMAC model | Same HMAC model |
| `idempotency_key` support | Yes | No |

## Migration Guidance

- **Stay on pusher-js / Echo** if you only need core Pusher behavior (V1 protocol).
- **Use `@sockudo/client`** to adopt protocol V2 features: connection recovery, serial numbers, message IDs, tag filtering, and delta compression.
- **Gradual migration**: the same Sockudo server handles both V1 and V2 connections simultaneously. Upgrade clients one at a time.
- **Server SDKs**: Sockudo server SDKs (`sockudo`, `sockudo-http-go`, etc.) are recommended for `idempotency_key` support. Pusher server SDKs still work for basic event triggering and channel auth.
