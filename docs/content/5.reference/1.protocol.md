## WebSocket Endpoint

Sockudo serves WebSocket upgrades on:

```text
/app/{appKey}
```

## Protocol Versioning

Sockudo supports two protocol versions, negotiated per-connection via the `?protocol=` query parameter:

| | Protocol V1 (default) | Protocol V2 |
|---|---|---|
| Query param | `?protocol=7` (or omitted) | `?protocol=2` |
| Event prefix | `pusher:` / `pusher_internal:` | `sockudo:` / `sockudo_internal:` |
| `serial` field | Never sent | Always on every message |
| `message_id` field | Never sent | Always on every broadcast |
| Connection recovery | Not available | Always available |
| Delta compression | Not available | Native |
| Tag filtering | Not available | Native |
| Message extras | Not available | Native |
| Echo control | Not available | Native |
| Event name filtering | Not available | Native |
| Ephemeral messages | Not available | Native |
| Compatible SDKs | Official Pusher SDKs (`pusher-js`) | Sockudo client SDKs (`@sockudo/client`) |

**V1 example URL** (Pusher-compatible):
```text
wss://host:6001/app/app-key?protocol=7&client=js&version=8.0.0
```

**V2 example URL** (Sockudo-native):
```text
wss://host:6001/app/app-key?protocol=2&echo_messages=false&client=sockudo-js&version=1.0.0
```

## V2 Wire Formats

Protocol V2 supports transport-level wire format negotiation through the `format` query parameter.

| Query param | Frame type | Notes |
|---|---|---|
| omitted / `json` | text | Default V2 format. |
| `messagepack` / `msgpack` | binary | Uses MessagePack encoding of the V2 message envelope. |
| `protobuf` / `proto` | binary | Uses protobuf encoding of the V2 message envelope. |

Examples:

```text
wss://host:6001/app/app-key?protocol=2&format=json
wss://host:6001/app/app-key?protocol=2&format=messagepack
wss://host:6001/app/app-key?protocol=2&format=protobuf
```

Notes:

- `format` is only honored for protocol V2.
- Protocol V1 always uses JSON text frames.
- Unknown V2 `format` values are rejected during the WebSocket handshake with HTTP `400 Bad Request`.
- The logical V2 message schema is the same across all formats; only the transport encoding changes.

Current SDK support for V2 wire formats in this workspace:

- `client-sdks/sockudo-js`
- `client-sdks/sockudo-flutter`
- `client-sdks/sockudo-kotlin`
- `client-sdks/sockudo-swift`

The server handles both protocol versions simultaneously on the same port. The version is per-connection — one server can serve Pusher-compatible and Sockudo-native clients at the same time.

## Extension Events (V2 only)

The following features are only available on protocol V2 connections:

### Delta Compression

- `sockudo:enable_delta_compression`
- `sockudo:delta_compression_enabled`
- `sockudo:delta`
- `sockudo:delta_cache_sync`
- `sockudo:delta_sync_error`

### Connection Recovery (V2 only)

Connection recovery events are only available on protocol V2 connections:

- `sockudo:resume` — sent by the client to request recovery of missed messages. Payload contains `channel_serials` mapping channel names to the last received serial number.
- `sockudo:resume_success` — sent by the server after processing a resume request. Payload contains `recovered` (channels with replayed messages) and `failed` (channels requiring re-subscription).
- `sockudo:resume_failed` — sent by the server per-channel when recovery is not possible (e.g., the requested serial has been evicted from the buffer). Payload contains `reason`.

## Extended Message Fields (V2 only)

Protocol V2 broadcast messages always include these additional fields:

| Field | Type | Description |
|---|---|---|
| `serial` | `u64` | Monotonically increasing sequence number per channel. |
| `message_id` | `string` | UUIDv4 identifier unique to this broadcast. |

These fields are never present on V1 connections.

### Message Extras Envelope (V2 only)

V2 broadcast messages may include an `extras` field containing per-message metadata. The `extras` object is optional and may contain any combination of the following fields:

| Field | Type | Description |
|---|---|---|
| `headers` | `object` | Flat key-value map of string metadata. Forwarded as-is to subscribers. Useful for server-side filtering or routing logic. |
| `ephemeral` | `bool` | When `true`, the message is not written to the recovery replay buffer and does not trigger webhooks. Useful for high-frequency updates (e.g., cursor positions) that do not need durability. |
| `idempotency_key` | `string` | Message-level deduplication key. The server uses an atomic SET NX operation scoped to `{app_id}:{channel}:{key}` to ensure only the first message with this key is broadcast. Uses the TTL from the `[idempotency]` server config. |
| `echo` | `bool` | Per-message echo control override. When `false`, the message is not sent back to the connection that published it, regardless of the connection-level `echo_messages` setting. |

#### V2 Broadcast with Extras Example

```json
{
  "event": "price-update",
  "channel": "market:BTC",
  "data": "{\"price\":42000}",
  "serial": 57,
  "message_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "extras": {
    "headers": { "source": "exchange-api", "region": "us-east" },
    "ephemeral": false,
    "idempotency_key": "price-btc-42000-v1",
    "echo": true
  }
}
```

## Subscribe Payload Extensions

V2 connections accept extra subscription fields:

- `filter` / `tags_filter`: filter tree for tag filtering.
- `filter.events`: array of event names for event name filtering. When set, only messages matching one of the listed event names are delivered to this subscription.
- `delta`: per-subscription delta settings.
- user-limited channel names: `namespace:channel#user_id`. These are only valid when the resolved namespace allows user-limited channels.
- wildcard channel patterns: a single `*` is supported for public-channel subscriptions only, for example `news.*` or `market-*-data`.
- metachannels: subscribe to `[meta]channel-name` to receive channel lifecycle events for `channel-name`.

V1 subscribe event: `pusher:subscribe`
V2 subscribe event: `sockudo:subscribe`

### V2 Metachannel Events

Metachannels are V2-only and use the `[meta]channel-name` convention.

Examples:

```text
[meta]orders
[meta]presence-room
```

Current metachannel events:

- `sockudo_internal:channel_occupied`
- `sockudo_internal:channel_vacated`
- `sockudo_internal:subscription_count`
- `sockudo_internal:member_added`
- `sockudo_internal:member_removed`

Example payload:

```json
{
  "event": "sockudo_internal:subscription_count",
  "channel": "[meta]orders",
  "data": {
    "channel": "orders",
    "subscription_count": 3
  }
}
```

## Interoperability Testing

The repository includes live cross-SDK V2 interoperability tests against a local Sockudo server.

Environment:

- `SOCKUDO_LIVE_TESTS=1` enables live tests.
- `SOCKUDO_WIRE_FORMAT=json|messagepack|protobuf` selects the V2 transport format under test.

Recommended local flow:

```bash
cargo run -p sockudo -- --config config/config.toml
```

In a second shell:

```bash
SOCKUDO_LIVE_TESTS=1 ./scripts/test-wire-format-matrix.sh
```

The matrix runner executes JS, Flutter, Kotlin, and Swift live tests for each V2 wire format.
