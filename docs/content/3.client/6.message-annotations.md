## Overview

Message annotations are a **Protocol V2 only** feature. V1 and Pusher-compatible clients do not receive annotation summary or raw annotation events.

The common client model is summary-first:

1. Subscribe to the channel normally.
2. When an annotation changes a message summary, Sockudo delivers a `message.summary` payload.
3. Merge the summary into local message state:

```js
message.annotations.summary[type] = summary;
```

There is no guarantee that a client sees every intermediate summary snapshot. Treat summaries as replace-by-type state, not as an append-only stream. Only the latest summary for each annotation type matters for normal UI.

On the V2 wire, summary payloads are delivered through Sockudo's internal message event with `data.action === "message.summary"`. SDKs may expose this as a `message.summary` event or as a normal channel event whose payload action is `message.summary`.

## Summary Shape

Summary payloads are keyed by the full annotation type:

```json
{
  "action": "message.summary",
  "serial": "00000000000000000001:node:00000000000000000001",
  "annotations": {
    "summary": {
      "reactions:distinct.v1": {
        "like": {
          "total": 2,
          "clientIds": ["user-1", "user-2"],
          "clipped": false
        }
      }
    }
  }
}
```

Merge by message serial and annotation type:

```js
function applySummary(localMessages, payload) {
  const message = localMessages.get(payload.serial) || {
    serial: payload.serial,
    annotations: { summary: {} },
  };

  message.annotations.summary = {
    ...message.annotations.summary,
    ...payload.annotations.summary,
  };

  localMessages.set(payload.serial, message);
}
```

If a summary bucket has `clipped: true`, the server omitted part of the contributor list to fit payload limits. The total remains authoritative; the client list is partial.

## Raw Annotation Streams

Raw annotation events are opt-in. Use them for audit trails, custom reducers, or raw annotation payloads. Do not use them as the default way to keep reaction counts or read-receipt counts current.

To receive raw `annotation.create` and `annotation.delete` events, subscribe with:

```json
{
  "modes": ["SUBSCRIBE", "ANNOTATION_SUBSCRIBE"]
}
```

Important: explicitly setting `modes` replaces default modes. If you set `modes`, include every mode you still need. Setting only `["ANNOTATION_SUBSCRIBE"]` can omit ordinary subscription behavior.

Raw annotation payloads arrive on the V2 internal annotation event with `data.action` equal to `annotation.create` or `annotation.delete`. Clients that consume raw events are responsible for maintaining their own local projection if they choose not to use `message.summary`.

## Reconnect Behavior

On reconnect, refresh the current summary state instead of replaying raw annotations from the last seen annotation serial.

Recommended approaches:

- resubscribe and use the summary snapshots delivered for existing channel projections
- fetch a fresh message/history snapshot through a trusted backend
- use a latest-summary HTTP surface when your deployment exposes one

Raw annotation replay is for precise audit and diagnostic views. It is not the right recovery mechanism for normal summary UI because missed intermediate summary snapshots are allowed.

## Examples

Runnable examples live in `examples/`:

- `examples/reactions_summary.js` subscribes normally and renders `reactions:distinct.v1` counts from `message.summary`
- `examples/read_receipts.js` publishes `receipts:flag.v1` annotations with identified `client_id` values and renders `total` plus `clientIds`
- `examples/moderation_flags.js` publishes `moderation:multiple.v1` annotations and renders per-client counts plus `totalUnidentified`

Run a local server with annotations enabled:

```bash
HISTORY_ENABLED=true VERSIONED_MESSAGES_ENABLED=true ANNOTATIONS_ENABLED=true cargo run -p sockudo
```

Then run an example:

```bash
node examples/reactions_summary.js
node examples/read_receipts.js
node examples/moderation_flags.js
```

The examples use `app-id`, `app-key`, and `app-secret` from the default local config. Override them with `SOCKUDO_APP_ID`, `SOCKUDO_APP_KEY`, and `SOCKUDO_APP_SECRET`.
