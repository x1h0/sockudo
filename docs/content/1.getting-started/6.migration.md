## Migration Strategy

Use phased rollout:

1. Keep your existing clients (pusher-js / Echo), point them to Sockudo host and stay on **Protocol V1**.
2. Move backend event publishing to Sockudo HTTP API or Pusher server SDK targeting Sockudo.
3. Validate auth endpoints and presence behavior.
4. Adopt Sockudo SDKs where you need Sockudo-specific capabilities.
5. Move selected clients to **Protocol V2** only when you want Sockudo-native features.

## Client Config Diff (Basic)

```ts [before-after.ts]
// before (hosted Pusher)
new Pusher("app-key", {
  cluster: "eu",
  forceTLS: true,
});

// after (Sockudo)
new Pusher("app-key", {
  cluster: "mt1",
  wsHost: "your.sockudo.host",
  wsPort: 6001,
  wssPort: 443,
  forceTLS: true,
  enabledTransports: ["ws", "wss"],
});
```

## Backend Trigger Diff

Most Pusher server SDKs can target Sockudo by setting `host` and `port`.

```ts
new Pusher({
  appId: process.env.PUSHER_APP_ID,
  key: process.env.PUSHER_APP_KEY,
  secret: process.env.PUSHER_APP_SECRET,
  host: "your.sockudo.host",
  port: 6001,
  useTLS: false,
});
```

## Compatibility Notes

- **Protocol V1** is Pusher-compatible and is the recommended migration starting point.
- **Protocol V2** is Sockudo-native and should be treated as an explicit product upgrade, not as a transparent compatibility mode.
- Signed HTTP API query parameters are required for `/apps/*` routes.
- Advanced tag filtering and delta decode are not native in stock pusher-js/Echo clients.
- Durable history, rewind, and continuity-aware recovery are V2-first features. Do not assume a V1 client can participate in them.

::note
For Sockudo-only features, migrate frontend clients to `@sockudo/client` incrementally, channel by channel if needed, and opt into Protocol V2 intentionally.
::

## Migrating Recovery

Older V2 recovery used replay-buffer-only positions:

```json
{
  "channel_serials": {
    "market:BTC": 54
  }
}
```

Current continuity-aware recovery should prefer:

```json
{
  "channel_positions": {
    "market:BTC": {
      "stream_id": "01HSZ7J0J4P8S7X3JCM4H9G2A1",
      "serial": 54,
      "last_message_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
    }
  }
}
```

Practical migration advice:

1. Keep V1 clients on the existing Pusher-compatible path.
2. Move V2 clients to tracking `stream_id`, `serial`, and `message_id` per channel.
3. Treat `sockudo:resume_failed` as a hard continuity failure, not as a soft warning.
4. Use the channel history HTTP API to rebuild client state when continuity cannot be proven.

## Failure Mode Changes

Replay-buffer-only recovery used to effectively mean:

- recover if the in-memory buffer still has the data
- otherwise resubscribe

Durable history plus continuity tokens changes that to:

- recover from the hot replay buffer when possible
- otherwise recover from durable history only when `stream_id` continuity still matches
- fail explicitly when continuity cannot be proven

This is a stricter contract by design. It avoids false-success recovery after:

- node restart
- replay buffer loss
- durable history retention gaps
- degraded durable-store conditions

## Presence History Migration

If you currently rely on live presence webhooks or your own persistence layer, treat Sockudo presence history as a complementary historical retrieval surface, not as a replacement for live presence state.

Keep these surfaces separate:

- live presence on subscribed channels:
  still provides immediate `member_added` / `member_removed` behavior and remains the Pusher-compatible path for V1 clients
- current member queries:
  use `GET /apps/{appId}/channels/{channelName}/users`
- historical presence events:
  use `GET /apps/{appId}/channels/{channelName}/presence/history`

Recommended migration path:

1. Keep existing live presence webhooks in place for real-time backend reactions.
2. Enable `presence_history` for the channels or namespaces where retained joins/leaves are useful.
3. Move operator or backend audit/reconciliation reads to `/presence/history` instead of scraping webhook archives.
4. Treat `/presence/history` as bounded retained event history, not as a complete user-presence database.
5. If `/presence/history` reports `degraded` or `reset_required`, fall back to your existing operational playbook and inspect `/presence/history/state` before trusting the stream again.

Practical limits:

- retention can truncate old presence events without implying an outage
- with the current `memory` backend, node restart clears retained presence-history rows on that node
- write failures can make continuity unprovable for an existing stream, in which case operators must call `/presence/history/reset` before the stream is considered healthy again

## Feature Rollout Guidance

Recommended order:

1. V1 compatibility rollout
2. V2 clients with `message_id`, `serial`, and `stream_id` tracking
3. Durable history API usage through your backend
4. Rewind on selected channels
5. Continuity-aware reconnect handling in production

## Recommended Rollout Checks

- `/up` and `/up/{appId}` healthy.
- Private and presence auth succeeds.
- Message fanout latency and ordering match expectations.
- Webhooks and metrics endpoints wired in production.
