# Push Notifications

Sockudo push notifications are optional and available only when the server is
built with the `push` feature plus the provider features you need. Push APIs are
HTTP/admin surfaces. They do not change the Pusher V1 WebSocket wire shape.

Release 4.5 push uses the durable queue-based pipeline from the binding capacity
model. The Publish API validates the request, persists the initial publish
status, appends the durable publish-log event, and only then returns
`202 Accepted` with `publishId`. It never synchronously fans out to FCM, APNs,
Web Push, HMS, or WNS.

## Configuration

Use the top-level `[push]` section. Memory storage and memory queue are for
tests and local development only.

```toml
[push]
storage_driver = "memory"
queue_driver = "memory"
fcm_enabled = false
apns_enabled = false
webpush_enabled = false
hms_enabled = false
wns_enabled = false
accept_worker_count = 1
planner_worker_count = 1
shard_worker_count = 1
dispatch_worker_count = 1
feedback_worker_count = 1
queue_partition_count = 1
channel_shard_count = 1
fanout_fast_threshold = 10000
fanout_shard_size = 100000
fanout_sync_threshold = 0
backpressure_lag_threshold_secs = 60
publish_status_ttl_days = 30
stale_device_max_age_days = 90
dry_run = false
analytics_enabled = false
analytics_retention_days = 30
scheduler_interval_secs = 5

[push.retry]
max_attempts = 5
initial_backoff_ms = 1000
max_backoff_ms = 60000
max_elapsed_secs = 86400
jitter = true
respect_retry_after = true

[push.circuit_breaker]
failure_threshold = 5
cooldown_secs = 60
half_open_max_inflight = 10

[push.default_quotas]
acceptance_rps = 100
delivery_quota_daily = 0
fanout_max = 0
inflight_max = 1000

[push.payload_redaction]
redact_payload = true
redact_template_data = true
redact_provider_overrides = true
allow_debug_payload_logging = false
```

Credential references are optional strings such as
`fcm_credential_ref = "secret://sockudo/push/fcm"`. If Sockudo stores provider
credential material, set `PUSH_CREDENTIAL_ENCRYPTION_KEY` before upload so the
server stores a decryptable AES-GCM envelope that workers can load later. If the
key is unset, local/dev uploads use a base64 plaintext envelope and should not
be used for production secrets. Do not put provider tokens, private keys, Web
Push auth secrets, or raw payload data in logs or metrics.

## Required Env Vars

The same release-prep controls can be provided with environment variables:

| Env var | Default | Purpose |
| --- | ---: | --- |
| `PUSH_FCM_ENABLED` | `false` | Enable FCM provider surfaces. |
| `PUSH_APNS_ENABLED` | `false` | Enable APNs provider surfaces. |
| `PUSH_WEBPUSH_ENABLED` | `false` | Enable Web Push provider surfaces. |
| `PUSH_HMS_ENABLED` | `false` | Enable HMS provider surfaces. |
| `PUSH_WNS_ENABLED` | `false` | Enable WNS provider surfaces. |
| `PUSH_CREDENTIAL_ENCRYPTION_KEY` | unset | Envelope encryption key reference or key material source. |
| `PUSH_FANOUT_FAST_THRESHOLD` | `10000` | Fanout below this uses fast-path planning. |
| `PUSH_FANOUT_SHARD_SIZE` | `100000` | Target recipient count per shard-path job. |
| `PUSH_FANOUT_SYNC_THRESHOLD` | `0` | Keep `0` for async-only Publish API behavior. |
| `PUSH_BACKPRESSURE_LAG_THRESHOLD_SECS` | `60` | Lag threshold used by push admission/backpressure policy. |
| `PUSH_PUBLISH_STATUS_TTL_DAYS` | `30` | Publish status retention. |
| `PUSH_FAILURE_THRESHOLD` | `5` | Circuit-breaker failure threshold. |
| `PUSH_SCHEDULER_INTERVAL_SECS` | `5` | Scheduled-publish polling interval. |
| `PUSH_STALE_DEVICE_MAX_AGE_DAYS` | `90` | Age threshold for stale-device cleanup. |
| `PUSH_ANALYTICS_ENABLED` | `false` | Enable bounded analytics rollups. |
| `PUSH_DEFAULT_ACCEPTANCE_RPS` | `100` | Default per-tenant publish acceptance rate. |
| `PUSH_DEFAULT_DELIVERY_QUOTA_DAILY` | `0` | Default daily delivery quota; `0` means unset. |
| `PUSH_DEFAULT_FANOUT_MAX` | `0` | Default fanout cap; `0` means unset. |
| `PUSH_DEFAULT_INFLIGHT_MAX` | `1000` | Default per-tenant in-flight cap. |

Existing compatibility aliases such as `PUSH_STORAGE_DRIVER`,
`PUSH_QUEUE_DRIVER`, `PUSH_FANOUT_MAX`, `PUSH_DELIVERY_QUOTA_DAILY`,
`PUSH_ACCEPTANCE_RATE_LIMIT`, and `PUSH_PUBLISH_LOG_MAX_LAG` are still honored by
the current HTTP admission layer.

## Authorization

Push requests use signed app API authentication and then apply an explicit push
capability:

- `push-admin`: app-scoped push administration, credential writes, direct
  publish, raw-recipient publish, registry listing, status, and worker feedback.
- `push-subscribe`: self-service device updates and channel push subscriptions
  for the authenticated device only.

App-signed requests default to `push-admin`. Restricted clients send:

```http
X-Sockudo-Push-Capability: push-subscribe
```

Direct activation through `push-admin` returns a `deviceIdentityToken`. Later
self-service updates authenticate with either:

```http
Authorization: Bearer <deviceIdentityToken>
```

or:

```http
X-Sockudo-Device-Identity-Token: <deviceIdentityToken>
```

## Publish Acceptance

`POST /apps/{app_id}/push/publish` and batch publish are asynchronous admission
surfaces. Batch results preserve caller item order. V2 `extras.push` can create a
server-side push publish event after normal channel publish authorization
succeeds; this is still a server-side HTTP/internal path and is not a V1
WebSocket shape change.

Fanout regimes are binding:

| Regime | Trigger | Behavior |
| --- | ---: | --- |
| Fast path | fanout `< PUSH_FANOUT_FAST_THRESHOLD` | Planner reads subscribers through paginated storage queries and emits provider batches directly. |
| Shard path | fanout `>= PUSH_FANOUT_FAST_THRESHOLD` | Planner emits resumable shard jobs. Shard workers stream bounded token pages and checkpoint progress. |

The shard path must never load every channel token into memory.

## Provider Dispatch

Provider dispatch runs only from provider-specific durable delivery queues.
Provider workers must enforce weighted-fair tenant scheduling, bounded
concurrency, provider `Retry-After`, circuit breakers, retry/DLQ state, and
provider-specific status classification.

Provider feedback must update device state, stale-token cleanup, publish status
counters, metrics, retry/DLQ state, and `[meta]log:push`.

Provider quota limits are external limits. Sockudo can smooth, defer, retry, and
surface throttling; it cannot force FCM, APNs, Web Push services, HMS, or WNS to
accept beyond their quotas or abuse controls.

## Backend Scale Tiers

Push backend scale claims follow the binding capacity model:

- memory: tests and development only.
- PostgreSQL/MySQL: small production, generally under 10M devices unless
  benchmarked.
- SurrealDB: small supported tier only unless benchmarked.
- DynamoDB: cloud-native production path.
- ScyllaDB: hyperscale recommended path.
- Redis: small production only unless Redis Streams reliability is implemented
  and tested.
- Kafka/Redpanda, Pulsar, NATS JetStream, Iggy, and Pub/Sub: high-scale durable
  queue candidates with documented limits.
- RabbitMQ/SQS: supported where ack/redelivery/visibility semantics fit.
- SNS: not a complete worker queue by itself; pair it with SQS or another
  pull-capable worker queue.
