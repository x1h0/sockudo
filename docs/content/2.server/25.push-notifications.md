# Push Notifications

Sockudo push notifications are optional and available only when the server is
built with the `push` feature. Push APIs are HTTP/admin surfaces; they do not
change the Pusher V1 WebSocket wire shape.

## Authorization

Push requests use the existing signed app API authentication and then apply an
explicit push capability:

- `push-admin`: full app-scoped push administration, credential writes, direct
  publish, raw-recipient publish, registry listing, removeWhere, status, and
  worker feedback APIs.
- `push-subscribe`: self-service device updates and channel push subscriptions
  for the authenticated device only.

App-signed requests default to `push-admin`. Clients that should be restricted
to self-service flows must send:

```http
X-Sockudo-Push-Capability: push-subscribe
```

Direct activation through `push-admin` returns a `deviceIdentityToken`. Store it
on the device side. Later self-service device updates and subscription changes
must authenticate with either:

```http
Authorization: Bearer <deviceIdentityToken>
```

or:

```http
X-Sockudo-Device-Identity-Token: <deviceIdentityToken>
```

Device identity tokens are stored as PBKDF2-SHA256 hashes. Rotate a token during
an admin device update with:

```http
X-Sockudo-Rotate-Device-Identity-Token: true
```

Deleting the device revokes the token.

## Publish Acceptance

`POST /apps/{app_id}/push/publish` and batch publish are asynchronous admission
surfaces. They validate, persist publish status, append a publish-log event, and
return `202 Accepted` with `publishId`. The Publish API does not fan out
synchronously to providers.

Publishing a realtime message with V2 `extras.push` creates a server-side push
publish event after normal channel publish authorization succeeds. No extra push
capability is required for that channel path.

Raw-recipient push publish targets require `push-admin`.

## Quotas And Backpressure

Current admission controls:

| Setting | Default | Behavior |
| --- | ---: | --- |
| `PUSH_DEFAULT_ACCEPTANCE_RPS` / `push.default_quotas.acceptance_rps` | `100` | Per-app Publish API acceptance requests per second, enforced through the configured Sockudo rate-limiter driver. `0` disables acceptance throttling. |
| `PUSH_ACCEPTANCE_RATE_LIMIT` | unset | Compatibility override for `PUSH_DEFAULT_ACCEPTANCE_RPS`. |
| `PUSH_FANOUT_MAX` | unset | If exceeded, the persisted publish status is `quota_exceeded`. |
| `PUSH_DELIVERY_QUOTA_DAILY` | unset | If exceeded by the accepted fanout estimate, status is `quota_exceeded`. |
| `PUSH_BACKPRESSURE` | `false` | When true, publish acceptance returns `503` with `Retry-After`. |
| `PUSH_BACKPRESSURE_RETRY_AFTER_SECONDS` | `5` | `Retry-After` value for backpressure responses. |
| `PUSH_PUBLISH_LOG_MAX_LAG` | `100000` | Maximum ready plus delayed publish-log depth accepted before the API returns `503`. |
| `PUSH_FANOUT_FAST_THRESHOLD` | `10000` | Fanout below this uses the fast-path regime. |
| `PUSH_FANOUT_SHARD_SIZE` | `100000` | Target recipient count for shard-path planning. |

Quota override is allowed only for `push-admin`:

```http
X-Sockudo-Push-Quota-Override: true
```

Credential writes, quota overrides, and removeWhere calls emit structured audit
logs under the `sockudo_push_audit` target. Credential writes also append
operator invalidation events so provider credential caches on other nodes can
reload through the cluster event path.

## Security

Provider credential responses never include credential material. Device registry
responses include provider, transport type, and token hash only; provider tokens,
Web Push endpoints, and device identity tokens are not returned.

Web Push endpoints are validated before storage:

- HTTPS is required outside local tests.
- `localhost` and private or local IP ranges are blocked unless
  `PUSH_ALLOW_LOCAL_WEB_PUSH_ENDPOINTS=true`.
- `PUSH_WEBPUSH_ALLOWED_HOSTS` and `PUSH_WEBPUSH_DENIED_HOSTS` accept
  comma-separated host names. Entries starting with `.` match suffixes.

Template substitution supports only bounded scalar placeholders such as
`{{ user.name }}` from `templateData`. Missing keys, object/array substitutions,
unterminated placeholders, and rendered fields over 4096 bytes are rejected.

`removeWhere` without a filter returns `400`. Filtered `removeWhere` scans the
registry with cursor pages and returns `"consistency": "eventual"` because
concurrent device registration or token rotation can race with cleanup.

## Backend Scale Tiers

Push backend scale claims follow the binding capacity model:

- memory: tests and development only.
- PostgreSQL/MySQL: small production, generally under 10M devices unless
  benchmarked. Runtime push stores are feature-gated behind `postgres` and
  `mysql`; checked-in migrations create the device registry, subscriptions,
  credentials, templates, schedules, publish status/log, fanout shard,
  idempotency, scheduler lock, delivery event, and operator invalidation tables.
- SurrealDB: small supported tier only unless benchmarked.
- DynamoDB: cloud-native production path.
- ScyllaDB: hyperscale recommended path.
- Redis: small production only unless Redis Streams reliability is implemented
  and tested.
- Kafka/Redpanda, Pulsar, NATS JetStream, Iggy, and Pub/Sub: high-scale durable
  queue candidates with documented operational limits.
- RabbitMQ/SQS: supported where ack/redelivery/visibility semantics fit.
- SNS: not a complete worker queue by itself.

Select runtime push storage with `push.storage_driver` or
`PUSH_STORAGE_DRIVER`. Supported runtime values are `memory`, `postgres`,
`mysql`, `dynamodb`, `surrealdb`, and `scylladb`.

Select runtime push queues with `push.queue_driver` or `PUSH_QUEUE_DRIVER`.
Supported runtime values are `memory`, `redis`, `redis-cluster`, `nats`,
`pulsar`, `rabbitmq`, `google-pubsub`, `kafka`, `iggy`, and `sqs` when the
matching feature is compiled. `sns` is not a worker queue by itself and remains
unsupported unless paired with a pull-capable queue.

## Provider Dispatch Runtime

Provider workers dispatch only from provider-specific durable delivery queues.
The Publish API never sends directly to FCM, APNs, Web Push, HMS, or WNS.

Provider clients are Sockudo wrappers with feature-gated HTTP transport:

- FCM uses HTTP v1 `projects/{project}/messages:send`, bearer auth supplied by
  a cached token provider, typed response classification, and `Retry-After`
  handling.
- APNs uses HTTP/2 request construction for `/3/device/{token}`, APNs topic,
  push type, priority, collapse ID, expiration headers, and APNs status/reason
  classification.
- Web Push validates HTTPS endpoints before dispatch and delegates RFC 8291
  encryption to the configured crypto adapter before HTTP send.
- HMS uses OAuth bearer auth, provider token list request shape, and per-token
  response classification.
- WNS uses OAuth bearer auth, channel URI POST, notification type validation,
  and WNS status/header classification.

Provider workers apply weighted-fair tenant scheduling before dispatch,
adaptive per-tenant/provider rate limiting, provider circuit breakers, and
retry/dead-letter routing from structured provider feedback. Over-quota tenants
are downgraded before provider dispatch, and each tenant lane has an in-flight
cap so one app cannot occupy every worker slot. Circuit breaker events are
emitted to `[meta]log:push` and provider outcome counters are exposed through
the push metrics abstraction.

Scheduled push workers poll due buckets from the shared store on every node,
take a per-job distributed lock, and emit publish-log events. They do not send
directly to providers.

### Provider Monolith Workers

Monolith dispatch workers are available for FCM, APNs, Web Push, HMS, and WNS
when the binary is built with `monolith` plus the matching `push-*` provider
feature and the corresponding `push.*_enabled` flag is true. FCM, HMS, and WNS
workers currently consume pre-generated bearer tokens from environment
variables (`PUSH_FCM_PROVIDER_TOKEN`, `PUSH_HMS_PROVIDER_TOKEN`, and
`PUSH_WNS_PROVIDER_TOKEN`). APNs can use either env token auth or a stored APNs
credential.

The `full` feature includes all provider features, but workers still require
their `push.*_enabled` config flag and provider-specific environment variables.

When the binary is built with `push-apns,monolith` and `push.apns_enabled` /
`PUSH_APNS_ENABLED=true`, Sockudo starts APNs dispatch workers in the monolith.
Prefer a stored APNs credential when the worker shares the push store:

```bash
export PUSH_CREDENTIAL_ENCRYPTION_KEY='change-me'

curl -X POST "$SOCKUDO_URL/apps/app-id/push/credentials/apns" \
  -H 'content-type: application/json' \
  -H 'x-sockudo-push-capability: push-admin' \
  -d '{
    "credentialId": "apns",
    "teamId": "TEAMID1234",
    "keyId": "KEYID12345",
    "privateKey": "-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----"
  }'

PUSH_APNS_TOPIC=com.example.app
PUSH_APNS_APP_ID=app-id
PUSH_APNS_CREDENTIAL_ID=apns
PUSH_APNS_ENDPOINT=https://api.sandbox.push.apple.com # sandbox only
```

The stored APNs credential can also use `pem`, or base64 `p12` plus optional
`p12Password`. Legacy APNs credentials uploaded before decryptable credential
envelopes must be re-uploaded because the old `sha256` envelope intentionally
cannot be decrypted by workers.

Without `PUSH_APNS_APP_ID`, configure env token auth with
`PUSH_APNS_TEAM_ID`, `PUSH_APNS_KEY_ID`, and `PUSH_APNS_PRIVATE_KEY_PATH` or
`PUSH_APNS_PRIVATE_KEY`. `PUSH_APNS_PROVIDER_TOKEN` may be supplied instead for
short-lived manual testing, but it is not rotated by Sockudo.

Feedback processing suppresses duplicate provider result events before updating
device state or publish counters. Duplicate suppression is metered through the
push metrics abstraction.

## Observability

Push emits structured logs for publish acceptance/completion, dispatch
success/failure, first delivery failure, device transition to `FAILED`, circuit
breaker changes, backpressure `503`, quota rejection, and credential writes or
load failures. Operator-facing push lifecycle events are emitted on the
`[meta]log:push` target for push accepted, publish completed, device state
changed, provider rejected, token invalidated, quota, scheduler, circuit
breaker, and dead-letter events.

The push metrics catalog uses fixed Prometheus names such as
`sockudo_push_publish_accepted_total`,
`sockudo_push_dispatch_duration_seconds`,
`sockudo_push_delivery_jobs_emitted_total`,
`sockudo_push_circuit_breaker_state`,
`sockudo_push_token_invalidations_total`, and
`sockudo_push_wfq_dispatched_total`. The Grafana dashboard template is checked
in at `ops/dashboards/push.json`.
