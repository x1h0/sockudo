## Overview

Channel rewind is a V2 subscribe-time option built on top of durable channel history.

Important boundary:

- `history` is an explicit HTTP fetch API
- `rewind` is a subscribe option that replays historical messages before normal live delivery resumes

Rewind does not replace the history API. It is an attach-time convenience feature.

## V2 Only

Rewind is only supported on protocol V2 subscriptions.

If a V1 client attempts to use rewind, Sockudo rejects the subscription request.

Guarantee level:

- Guaranteed: subscribe-time request shape, `sockudo:rewind_complete` payload, local delivery ordering.
- Best-effort: completeness when the retained history floor or server-side page cap truncates the request.
- Intentionally unsupported: rewind on protocol V1.

## Subscribe Shape

Sockudo accepts rewind inside the subscribe payload:

Count-based rewind:

```json
{
  "event": "sockudo:subscribe",
  "data": {
    "channel": "market:BTC",
    "rewind": {
      "count": 25
    }
  }
}
```

Time-based rewind:

```json
{
  "event": "sockudo:subscribe",
  "data": {
    "channel": "market:BTC",
    "rewind": {
      "seconds": 60
    }
  }
}
```

Sockudo also accepts a shorthand integer:

```json
{
  "event": "sockudo:subscribe",
  "data": {
    "channel": "market:BTC",
    "rewind": 10
  }
}
```

Official SDKs expose rewind at subscribe time and surface `sockudo:rewind_complete` on the subscribed channel:

::code-group

```js [JavaScript]
const channel = client.subscribe("market:BTC", {
  rewind: { seconds: 60 },
});

channel.bind("sockudo:rewind_complete", (payload) => {
  console.log(payload.historical_count, payload.live_count, payload.complete);
});
```

```swift [Swift]
let channel = client.subscribe(
    "market:BTC",
    options: .init(rewind: .seconds(60))
)

channel.bind("sockudo:rewind_complete") { data, _ in
    print(data as Any)
}
```

```kotlin [Kotlin]
val channel = client.subscribe(
    "market:BTC",
    SubscriptionOptions(rewind = SubscriptionRewind.Seconds(60))
)

channel.bind("sockudo:rewind_complete") { data, _ ->
    println(data)
}
```

```dart [Flutter]
final channel = client.subscribe(
  'market:BTC',
  options: const SubscriptionOptions(
    rewind: SubscriptionRewind.seconds(60),
  ),
);

channel.bind('sockudo:rewind_complete', (data, _) {
  print(data);
});
```

::

## Delivery Order

For rewinded subscriptions, Sockudo guarantees this local handoff order:

1. the subscription succeeds
2. historical messages are replayed oldest-to-newest
3. live messages that arrived during the rewind window are flushed
4. Sockudo emits `sockudo:rewind_complete`

This keeps the handoff gap-free on the local delivery path.

## Duplicate Handling

Sockudo suppresses duplicates during the rewind handoff using:

- channel `serial`
- `message_id` when present

If a message appears in both:

- the rewinded history slice
- and the live buffer accumulated during the handoff

Sockudo sends it once.

## Completion Event

After rewind finishes, Sockudo sends:

```json
{
  "event": "sockudo:rewind_complete",
  "channel": "market:BTC",
  "data": {
    "historical_count": 25,
    "live_count": 2,
    "complete": true,
    "truncated_by_retention": false,
    "truncated_by_limit": false
  }
}
```

Meanings:

- `historical_count`: number of rewinded history messages delivered
- `live_count`: number of buffered live messages flushed after the history slice
- `complete`: whether Sockudo fully satisfied the rewind request
- `truncated_by_retention`: older requested data had already expired from retention
- `truncated_by_limit`: the request exceeded the server-side rewind/history page cap

## Guarantees And Limits

Guaranteed:

- rewind and live handoff remain separate from the explicit history API
- historical replay is delivered in chronological order
- live messages published during the handoff are buffered and flushed after history
- duplicate messages from the handoff window are suppressed locally

Best-effort / current limits:

- rewind uses the current durable history page limit as its cap
- time-based rewind is limited by retained durable history and server page limits
- multinode correctness hardening is still a later phase, so this document describes the current local handoff guarantee, not a final global guarantee under every distributed failure mode

Direct client support vs backend proxy:

- Official Sockudo client SDKs support subscribe-time rewind directly over WebSocket.
- Older data beyond the rewind slice still requires the backend-facing HTTP history API.
