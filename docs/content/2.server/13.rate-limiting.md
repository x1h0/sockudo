Sockudo provides three independent rate limiting layers, each targeting a different threat vector:

| Layer | Where | Scope | On Exceed |
|---|---|---|---|
| **Global (IP)** | HTTP middleware, before upgrade | Per client IP | HTTP 429 |
| **Message** | Inside WS loop, every message | Per connection | `pusher:error` or WS close |
| **Client events** | Inside WS loop, `client-*` only | Per connection | `pusher:error` or WS close |

## Global Rate Limiter

Protects the server door. Enforced by IP address before any WebSocket upgrade or HTTP API call completes.

Two independent limits are enforced:

| Limit | Protects | Default |
|---|---|---|
| **API rate limit** | HTTP API endpoints (`/apps/{appId}/events`, etc.) | 100 requests / 60 s |
| **WebSocket rate limit** | New WebSocket connections (`/app/{appKey}`) | 20 connections / 60 s |

### Configuration

```toml
[rate_limiter]
enabled = true
driver = "memory"

[rate_limiter.api_rate_limit]
max_requests = 100
window_seconds = 60
trust_hops = 0

[rate_limiter.websocket_rate_limit]
max_requests = 20
window_seconds = 60
trust_hops = 0
```

| Setting | Env Var | Default | Description |
|---|---|---|---|
| `enabled` | `RATE_LIMITER_ENABLED` | `true` | Enable/disable globally |
| `driver` | `RATE_LIMITER_DRIVER` | `memory` | Backend: `memory`, `redis`, `redis-cluster`, `none` |
| `api_rate_limit.max_requests` | `RATE_LIMITER_API_MAX_REQUESTS` | `100` | Max API requests per window per IP |
| `api_rate_limit.window_seconds` | `RATE_LIMITER_API_WINDOW_SECONDS` | `60` | Window duration in seconds |
| `api_rate_limit.trust_hops` | `RATE_LIMITER_API_TRUST_HOPS` | `0` | Trusted proxy hops for `X-Forwarded-For` |
| `websocket_rate_limit.max_requests` | `RATE_LIMITER_WS_MAX_REQUESTS` | `20` | Max WS connections per window per IP |
| `websocket_rate_limit.window_seconds` | `RATE_LIMITER_WS_WINDOW_SECONDS` | `60` | Window duration in seconds |

### Drivers

| Driver | Shared Across Nodes | External Dependency |
|---|---|---|
| `memory` | No | None |
| `redis` | Yes | Redis 6+ |
| `redis-cluster` | Yes | Redis Cluster 6+ |
| `none` | N/A | None |

The `memory` driver is node-local â€” each Sockudo instance tracks limits independently. For multi-node deployments use `redis` so limits are enforced cluster-wide.

### Proxy Trust (`trust_hops`)

When running behind a reverse proxy, client IPs appear as the proxy's IP. `trust_hops` controls how many `X-Forwarded-For` hops to trust:

| Value | Behavior |
|---|---|
| `0` | Use direct connection IP (default, safest) |
| `1` | Trust one proxy hop (single load balancer) |
| `2` | Trust two hops (CDN + load balancer) |

::callout{type="warning"}
Setting `trust_hops` too high lets clients spoof their IP. Only trust the exact number of proxies in your infrastructure.
::

### Response Headers

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1707744060
Retry-After: 45   # only on 429
```

## Per-App Message Rate Limit

Limits the total number of inbound WebSocket messages on a single connection, regardless of event type. This protects against connected clients hammering the server with any combination of `pusher:subscribe`, `pusher:ping`, `client-*`, or other events.

Configured per-app in `policy.limits.message_rate_limit`.

```toml
[app_manager.array.apps.policy.limits.message_rate_limit]
enabled = true
max_attempts = 60      # messages allowed within the window
decay_seconds = 60     # sliding window size in seconds
terminate_on_limit = false  # true = close connection, false = send pusher:error
```

| Field | Default | Description |
|---|---|---|
| `enabled` | `false` | Enable this limiter for the app |
| `max_attempts` | `60` | Messages allowed within `decay_seconds` |
| `decay_seconds` | `60` | Sliding window size in seconds |
| `terminate_on_limit` | `false` | If `true`, close the connection with code `4301`; if `false`, send `pusher:error` and keep the connection open |

When `terminate_on_limit = false` the client receives:

```json
{ "event": "pusher:error", "data": { "code": 4301, "message": "Client event rejected due to rate limit" } }
```

When `terminate_on_limit = true` the connection is closed with WebSocket close code `4301`.

::callout{type="info"}
This limiter is disabled by default (`enabled = false`). Enable it explicitly per-app.
::

## Per-App Client Event Rate Limit

Limits only `client-*` events (peer-to-peer broadcasts sent by connected clients). This runs after the message rate limit and applies regardless of whether the message rate limit is enabled.

Configured via `max_client_events_per_second` in `policy.limits`:

```toml
[app_manager.array.apps.policy.limits]
max_client_events_per_second = 10   # events allowed within decay_seconds
decay_seconds = 1                   # sliding window size in seconds (default: 1)
terminate_on_limit = false          # true = close connection on exceed
```

| Field | Default | Description |
|---|---|---|
| `max_client_events_per_second` | `0` (disabled) | Client events allowed within the window |
| `decay_seconds` | `1` | Sliding window size in seconds |
| `terminate_on_limit` | `false` | If `true`, close connection with code `4301` on exceed |

Setting `max_client_events_per_second = 0` disables the client event rate limiter entirely for that app.

::callout{type="info"}
The name `max_client_events_per_second` is historical. With `decay_seconds = 60` it means "max client events per 60 seconds", not per second.
::

## Choosing the Right Layer

| Scenario | Use |
|---|---|
| Prevent connection floods from a single IP | Global rate limiter |
| Prevent a connected client from spamming any message | `message_rate_limit` |
| Prevent clients from abusing peer-to-peer broadcasts | `max_client_events_per_second` |
| Both of the above | Enable both per-app limiters |
