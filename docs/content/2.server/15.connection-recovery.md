## Overview

Connection recovery allows clients to resume a WebSocket session after a brief disconnect and receive any messages they missed while offline. This feature is **protocol V2 only** and must be **enabled in config**. V1 (Pusher-compatible) connections never receive serial numbers or recovery events regardless of config.

Guarantee level:

- Guaranteed: canonical request/response event names and structured payload fields.
- Best-effort: hot recovery from the node-local replay buffer.
- Best-effort: cold recovery from durable history, subject to retained floor and backend availability.
- Intentionally unsupported: continuity-aware recovery on protocol V1.

This feature is useful for:

- Mobile applications with intermittent connectivity
- Users switching between Wi-Fi and cellular networks
- Brief server-side restarts or load balancer re-routing
- Any scenario where sub-second to two-minute disconnects are expected

## Configuration

Connection recovery requires protocol V2 (`?protocol=2`) AND must be enabled in config:

```toml [config/config.toml]
[connection_recovery]
enabled = true
buffer_ttl_seconds = 120
max_buffer_size = 100
```

Or via environment variables:

```bash
CONNECTION_RECOVERY_ENABLED=true
CONNECTION_RECOVERY_BUFFER_TTL=120
CONNECTION_RECOVERY_MAX_BUFFER_SIZE=100
```

| Field | Type | Default | Description |
|---|---|---|---|
| `enabled` | `bool` | `false` | Enable the connection recovery subsystem (V2 only) |
| `buffer_ttl_seconds` | `u64` | `120` | How long (in seconds) to retain buffered messages per channel |
| `max_buffer_size` | `usize` | `100` | Maximum number of messages to buffer per channel |

When enabled, V2 connections get:
- `serial` on every broadcast message
- `stream_id` on durable-history-backed V2 broadcasts
- The replay buffer for `sockudo:resume` recovery
- Clients can send `sockudo:resume` to recover missed messages

V1 connections never receive `serial` fields and cannot use recovery, regardless of this setting.

### Message IDs

When connection recovery is enabled, every V2 broadcast message automatically includes a unique `message_id` (UUIDv4). No separate config flag is needed — `message_id` is bundled with recovery.

## How It Works

### Serial Numbers

Sockudo assigns a monotonically increasing **serial number** to every message broadcast on a channel for V2 connections. The `serial` field is a `u64` value that starts at 1 and increments for each message on that channel. Clients use these serial numbers to track which messages they have received.

A V2 broadcast message:

```json
{
  "event": "price-update",
  "channel": "market:BTC",
  "data": "{\"price\":42000}",
  "stream_id": "01HSZ7J0J4P8S7X3JCM4H9G2A1",
  "serial": 57,
  "message_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

### Message Buffering

The server maintains a per-channel ring buffer of recent messages. The buffer is bounded by two constraints:

1. **Time**: messages older than `buffer_ttl_seconds` are evicted.
2. **Count**: at most `max_buffer_size` messages are retained per channel.

When either limit is exceeded, the oldest messages are dropped.

### Resume Protocol

When a client reconnects, it can request recovery of missed messages:

#### 1. Client Sends `sockudo:resume`

Preferred V2 shape:

```json
{
  "event": "sockudo:resume",
  "data": {
    "channel_positions": {
      "market:BTC": {
        "stream_id": "01HSZ7J0J4P8S7X3JCM4H9G2A1",
        "serial": 54,
        "last_message_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
      }
    }
  }
}
```

This is the canonical public request form.

Legacy compatibility is still accepted for the hot replay path only:

```json
{
  "event": "sockudo:resume",
  "data": {
    "channel_serials": {
      "market:BTC": 54
    }
  }
}
```

`channel_positions` is the continuity-aware form. Sockudo uses:

- `stream_id` to detect resets and loss of proof
- `serial` to request messages after the last confirmed position
- `last_message_id` as an optional dedup hint

#### 2. Server Replays Missed Messages

For each channel where recovery succeeds, the server sends the missed messages in order, each with their original `serial` and `stream_id` when present.

For channels where recovery is not possible, the server sends a per-channel structured failure:

```json
{
  "event": "sockudo:resume_failed",
  "channel": "market:ETH",
  "data": {
    "code": "position_expired",
    "reason": "retained_history_floor_is_ahead_of_requested_serial",
    "expected_stream_id": "01HSZ7J0J4P8S7X3JCM4H9G2A1",
    "current_stream_id": "01HSZ7J0J4P8S7X3JCM4H9G2A1",
    "oldest_available_serial": 80,
    "newest_available_serial": 120
  }
}
```

Common failure codes:

- `continuity_unverifiable`
- `stream_reset`
- `position_expired`
- `persistence_unavailable`

The client should re-subscribe to any channel that receives `sockudo:resume_failed`.

#### 3. Server Sends `sockudo:resume_success`

After processing all channels, the server sends a summary:

```json
{
  "event": "sockudo:resume_success",
  "data": {
    "recovered": [
      {
        "channel": "market:BTC",
        "source": "hot",
        "replayed": 3
      }
    ],
    "failed": [
      {
        "channel": "market:ETH",
        "code": "position_expired",
        "reason": "retained_history_floor_is_ahead_of_requested_serial"
      }
    ]
  }
}
```

`source` is `hot` when the replay buffer satisfied the request and `cold` when Sockudo fell back to durable history.

Direct client support vs backend proxy:

- The WebSocket `sockudo:resume` event is sent directly by official Sockudo client SDKs.
- Refreshing full state after `sockudo:resume_failed` typically requires a backend proxy or other trusted server-side use of the HTTP history API.

## Message Deduplication

### Broadcast `message_id`

Every V2 broadcast message includes a unique `message_id` (UUIDv4). Clients can use this to deduplicate messages if they receive the same message through both a live subscription and recovery replay.

### HTTP API Idempotency

The HTTP API supports an `idempotency_key` field on event trigger requests. When provided, the server ensures the event is only broadcast once, even if the HTTP request is retried:

```bash
curl -X POST http://localhost:6001/apps/app-id/events \
  -H "Content-Type: application/json" \
  -d '{
    "name": "price-update",
    "channel": "market:BTC",
    "data": "{\"price\":42000}",
    "idempotency_key": "unique-request-id-123"
  }'
```

The server caches the `idempotency_key` and returns the same response for duplicate requests within the buffer TTL window.

## Client SDK Usage

::code-group

```js [JavaScript]
import Sockudo from "@sockudo/client";

const client = new Sockudo("app-key", {
  wsHost: "127.0.0.1",
  wsPort: 6001,
  forceTLS: false,
  protocolVersion: 2,
  connectionRecovery: true,
});

const channel = client.subscribe("market:BTC");
channel.bind("message", () => {
  console.log(client.getRecoveryPosition("market:BTC"));
});

client.bind("sockudo:resume_success", (payload) => {
  console.log(payload.recovered);
});
```

let client = try SockudoClient(

}

client.connect()

## Limitations

- **Per-node buffer**: message buffers are local to each server node. If a client reconnects to a different node in a cluster, recovery will not have access to messages buffered on the original node.
- **Buffer window**: the default 120-second TTL means messages older than 2 minutes cannot be recovered. Increase `buffer_ttl_seconds` if your use case requires a longer window, at the cost of higher memory usage.
- **Memory usage**: each buffered message is held in memory. With many active channels and large `max_buffer_size` values, memory consumption can grow significantly.
- **No persistence**: message buffers are not persisted to disk. A full server restart clears all buffers.
- **V2 only**: V1 (Pusher-compatible) connections do not support recovery.

## Example Configuration

```toml [config/config.toml]
[connection_recovery]
buffer_ttl_seconds = 120
max_buffer_size = 100

[event_limits]
max_channels_at_once = 100
max_name_length = 200
max_payload_in_kb = 100
max_batch_size = 10
```

::callout{type="tip"}
`connection_recovery.enabled` defaults to `false`. Enable it in your config to activate connection recovery for V2 connections. V1 connections are never affected.
