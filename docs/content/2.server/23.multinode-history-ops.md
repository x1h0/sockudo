## Source Of Truth

For V2 durable history channels, the source of truth for:

- `stream_id`
- `serial`
- retained history floor and ceiling

is the durable history store, not any individual node-local replay buffer.

In practice:

- the publish node reserves the next `(stream_id, serial)` from the durable store
- that stamped message is fanned out locally and then broadcast to other nodes
- remote nodes deliver the origin node's serialized message and do not allocate a second serial

This prevents per-node serial drift under horizontal fanout.

Guarantee level:

- Guaranteed: durable history is the source of truth for retained `stream_id` and `serial`.
- Best-effort: node-local hot replay remains available only while the reconnect lands on a node with matching buffered evidence.
- Intentionally unsupported: distributed-consensus guarantees for every network partition shape.

## Durable Backend Scope

Phase-3 product stance:

- PostgreSQL, MySQL, ScyllaDB, SurrealDB, and DynamoDB are the current durable production history backends.
- `memory` remains useful for local development and tests, but not as a production continuity store.

ScyllaDB-specific prerequisite:

- the history keyspace must be created with `tablets = {'enabled': false}` because Sockudo uses LWT on the authoritative stream row for serial reservation
- when Sockudo creates the history keyspace itself, it applies that setting automatically
- if operators point Sockudo at an existing tablet-enabled keyspace, history initialization succeeds but serial reservation fails closed with an explicit configuration error

SurrealDB-specific note:

- the current durable history implementation uses optimistic compare-and-set reservation on the per-channel stream record plus deterministic entry record IDs
- retained floor/ceiling metadata is now persisted on the stream record, so inspection and recovery gating no longer need a full retained-entry scan
- this preserves correctness and passes the shared conformance suite, but it is not yet as read-optimized for very large streams as the PostgreSQL and MySQL paths

DynamoDB-specific note:

- the current durable history implementation uses one authoritative stream item plus one ordered item partition per `(app_id, channel, stream_id)`
- serial reservation relies on conditional writes, and retained history scans are partition-local
- retained floor/ceiling metadata is now persisted on the stream item, so inspection and recovery gating no longer need a full partition scan
- this preserves correctness and conformance, but it is not yet as read-efficient for very large retained windows as the PostgreSQL and MySQL paths

## Node Restart

On node restart:

- the local replay buffer is lost
- local hot recovery is no longer available for messages that only existed in memory on that node
- continuity-aware recovery falls back to durable history when the client's `stream_id` still matches and the retained floor still covers the requested serial

Operators should expect:

- more `cold` recovery successes after restarts
- fewer `hot` recovery successes until the replay buffer warms again

## Durable-Store Outage

When the durable history writer queue overflows or a durable append fails:

- the affected channel's durable stream row is marked `degraded`
- that durable row is now the authoritative cross-node source of truth for whether cold history/recovery is allowed
- a timestamped degraded hint is also written to local memory and shared cache when available
- history reads and cold recovery fail for that channel instead of pretending continuity is still guaranteed

This is intentional. Recovery correctness is preferred over silent data loss.

Important phase-3 behavior:

- degraded state is sticky until an explicit operator repair or reset flow clears or rotates it
- later successful appends do not silently re-enable cold recovery
- cache disagreement is resolved fail-closed: a newer degraded cache hint blocks recovery until the durable row catches up or is intentionally repaired

## Replay Buffer Loss

Replay buffer loss alone does not imply history loss.

If the durable store is healthy and the client presents a matching `stream_id`, Sockudo can still recover from durable history.

If the client only presents legacy `channel_serials`, Sockudo can only trust the hot replay buffer. Once that evidence is gone, recovery fails with `continuity_unverifiable`.

## Retention Compaction

Retention compaction advances the oldest retained serial and timestamp.

When a client asks for a position older than that retained floor:

- history reads report `truncated_by_retention`
- recovery fails with `position_expired`

This is not treated as a stream reset. It is a normal retention boundary.

## Metrics To Watch

Critical signals:

- `history_write_failures_total`
- `history_reset_required_channels`
- `history_degraded_channels`
- `history_queue_depth`
- `history_recovery_failures_total{code=...}`
- `history_recovery_success_total{source=\"hot\"}`
- `history_recovery_success_total{source=\"cold\"}`

Additional useful gauges:

- `history_reset_required_channels`
- `history_retained_messages`
- `history_retained_bytes`
- `history_write_latency_ms`

Healthy steady-state expectations:

- `history_degraded_channels = 0`
- `history_reset_required_channels = 0`
- queue depth stays near zero or drains quickly after bursts
- most reconnects after brief disconnects recover from `hot`
- `cold` recoveries rise during restarts or node rebalancing

## SLOs

Recommended production SLOs for durable-history-enabled applications:

| Area | SLO target | Measurement |
| --- | --- | --- |
| History write success | `>= 99.99%` successful writes over 30 days | `history_writes_total` vs `history_write_failures_total` |
| Persistence-backed recovery success | `>= 99%` over 30 days | `history_recovery_success_total` versus persistence-related recovery failures |
| Queue health | queue depth drains back to `0` quickly; sustained non-zero queue depth is abnormal | `history_queue_depth` |
| Degraded channels | `0` degraded channels in steady state | `history_degraded_channels` |
| Reset-required channels | `0` reset-required channels in steady state | `history_reset_required_channels` |

Notes:

- Recovery SLOs should focus on persistence-caused failures. Client-caused outcomes such as `position_expired`, `continuity_unverifiable`, or operator-induced `stream_reset` are useful operational counters, but they are not platform-availability failures.
- Queue depth is better treated as a saturation SLI than a success-rate SLI.

## `/stats` Surface

`GET /stats` now includes a `history` block:

```json
{
  "history": {
    "enabled": true,
    "backend": "postgres",
    "state_authority": "durable_store",
    "degraded_channels": 0,
    "reset_required_channels": 0,
    "queue_depth": 0
  }
}
```

This is a coarse operator summary:

- `state_authority` identifies the source of truth used for clustered recovery decisions
- `degraded_channels > 0` means at least one channel should not be trusted for cold history/recovery
- `reset_required_channels > 0` means at least one stream requires an explicit reset before cold continuity can resume
- sustained `queue_depth > 0` means the durable writer is falling behind

Per-channel inspection is available through:

- `GET /apps/{appId}/channels/{channelName}/history/state`
- `POST /apps/{appId}/channels/{channelName}/history/reset`
- `POST /apps/{appId}/channels/{channelName}/history/purge`

The inspection endpoint reports the current `stream_id`, `next_serial`, retained floor/ceiling, channel `durable_state`, whether `recovery_allowed`, whether `reset_required`, the last transition timestamp, and the observed hint source when a newer cache/local degraded hint is forcing fail-closed behavior.

Operator controls:

- `history/reset` is intentionally destructive: it purges retained history and rotates `stream_id`
- `history/purge` trims retained rows without rotating `stream_id`
- both endpoints require normal signed app API authentication and explicit confirmation fields in the JSON body
- both operations are logged with channel, reason, and optional `requested_by`

## Practical Alerting

Recommended alerts:

- page when `history_degraded_channels > 0` for more than 5 minutes
- page when `history_reset_required_channels > 0` for more than 2 minutes
- warn when `history_queue_depth > 0` for 10 minutes
- page when `history_queue_depth > 100` for 5 minutes
- warn when durable history write failure ratio exceeds `0.1%` over 10 minutes
- page when durable history write failure ratio exceeds `1%` over 5 minutes
- warn on persistence-related recovery failures over 10 minutes
- page when persistence-related recovery failure ratio exceeds `5%` over 15 minutes
- investigate when `cold` recoveries spike without a planned restart or rebalance

Severity guidance:

- `critical` / paging:
  degraded channels, reset-required channels, sustained critical queue depth, high write failure ratio, high persistence-related recovery failure ratio
- `warning`:
  queue not draining, SLO burn before breach, persistence-related recovery failures with low volume, unusual cold-recovery volume changes

## Distributed Decision Model

The durable `*_streams` row is authoritative for:

- `stream_id`
- retained floor and ceiling
- `durable_state` (`healthy`, `degraded`, `reset_required`)
- whether cold history/recovery is allowed

Shared cache is now a secondary propagation hint only.

- If durable metadata is healthy and no newer degraded hint exists, cold recovery may proceed.
- If durable metadata is `degraded`, all nodes must deny cold recovery.
- If durable metadata is `reset_required`, all nodes must deny cold recovery and operators should rotate/reset the stream.
- If a newer degraded hint exists than the durable row, nodes fail closed until the durable row reflects that transition or an operator repair supersedes it.

This is stronger than the phase-2 best-effort cache marker, but it is still not full distributed consensus for every partition shape. The design deliberately prefers false-negative recovery over split-brain false success.

## Reset And Purge Warnings

Use `history/reset` when continuity is intentionally broken.

- Sockudo rotates `stream_id`
- retained history is purged
- reconnecting clients on the old stream fail with `stream_reset`
- future publishes start on a new stream from serial `1`

Use `history/purge` when continuity is still valid but retained backlog must be trimmed.

- Sockudo keeps the current `stream_id`
- reconnecting clients before the new floor fail with `position_expired`
- clients at or after the retained floor can still use cold recovery

These controls are not generic admin primitives. They are explicit per-channel actions with narrow semantics so operators can reason about downstream client impact before pulling the trigger.

## Fault Matrix

Targeted failure-injection coverage maps faults to explicit outcomes:

| Fault | Expected behavior | Client-visible outcome |
| --- | --- | --- |
| writer queue full on publish | reject publish before live fanout | publish fails; no live delivery for that message |
| durable write failure after enqueue | mark stream `degraded` and fail cold history/recovery closed | later rewind/history/recovery fail with persistence-related errors until repaired or reset |
| cache marker read/state lookup failure | fail closed instead of assuming healthy continuity | recovery fails with `persistence_unavailable` |
| node restart / replay buffer loss | lose hot path only; durable history remains authoritative | reconnect falls back to `cold` recovery when retained floor still covers the requested serial |
| retention boundary crossed during reconnect | do not continue replay from an unverifiable cursor | recovery fails with `position_expired` |
| stream epoch changed during reconnect | do not mix old cursor with new stream | recovery fails with `stream_reset` |

Remaining blind spots:

- cache marker write failure is protected by the durable row being authoritative, but there is no dedicated live Postgres test that injects a cache `set` failure during degradation transition
- durable write failure is covered through explicit degraded-state behavior, not by a live Postgres faulting backend in CI
- horizontal transport publish failure itself is still covered more strongly by adapter transport tests than by the history-specific handler lane
