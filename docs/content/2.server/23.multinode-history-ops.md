## Source Of Truth

For V2 durable history channels, the source of truth for:

- `stream_id`
- `serial`
- retained history floor and ceiling

is the durable history store, not any individual node-local replay buffer.

In practice:

- the publish node reserves the next `(stream_id, serial)` from the durable store
- that stamped message is fanned out locally and then broadcast to other nodes
- remote nodes deliver the origin node's serialized message and do not allocate a second serial

This prevents per-node serial drift under horizontal fanout.

Guarantee level:

- Guaranteed: durable history is the source of truth for retained `stream_id` and `serial`.
- Best-effort: node-local hot replay remains available only while the reconnect lands on a node with matching buffered evidence.
- Intentionally unsupported: distributed-consensus guarantees for every network partition shape.

## Node Restart

On node restart:

- the local replay buffer is lost
- local hot recovery is no longer available for messages that only existed in memory on that node
- continuity-aware recovery falls back to durable history when the client's `stream_id` still matches and the retained floor still covers the requested serial

Operators should expect:

- more `cold` recovery successes after restarts
- fewer `hot` recovery successes until the replay buffer warms again

## Durable-Store Outage

When the durable history writer queue overflows or a durable append fails:

- the affected channel is marked degraded
- a degraded marker is also written to the shared cache when available
- history reads and cold recovery fail for that channel instead of pretending continuity is still guaranteed

This is intentional. Recovery correctness is preferred over silent data loss.

## Replay Buffer Loss

Replay buffer loss alone does not imply history loss.

If the durable store is healthy and the client presents a matching `stream_id`, Sockudo can still recover from durable history.

If the client only presents legacy `channel_serials`, Sockudo can only trust the hot replay buffer. Once that evidence is gone, recovery fails with `continuity_unverifiable`.

## Retention Compaction

Retention compaction advances the oldest retained serial and timestamp.

When a client asks for a position older than that retained floor:

- history reads report `truncated_by_retention`
- recovery fails with `position_expired`

This is not treated as a stream reset. It is a normal retention boundary.

## Metrics To Watch

Critical signals:

- `history_write_failures_total`
- `history_degraded_channels`
- `history_queue_depth`
- `history_recovery_failures_total{code=...}`
- `history_recovery_success_total{source=\"hot\"}`
- `history_recovery_success_total{source=\"cold\"}`

Healthy steady-state expectations:

- `history_degraded_channels = 0`
- queue depth stays near zero or drains quickly after bursts
- most reconnects after brief disconnects recover from `hot`
- `cold` recoveries rise during restarts or node rebalancing

## `/stats` Surface

`GET /stats` now includes a `history` block:

```json
{
  "history": {
    "enabled": true,
    "backend": "postgres",
    "degraded_channels": 0,
    "queue_depth": 0
  }
}
```

This is a coarse operator summary:

- `degraded_channels > 0` means at least one channel should not be trusted for cold history/recovery
- sustained `queue_depth > 0` means the durable writer is falling behind

## Practical Alerting

Recommended alerts:

- page when `history_degraded_channels > 0` for more than a short grace window
- warn when `history_queue_depth` stays elevated across several intervals
- warn on rising `history_recovery_failures_total`
- investigate when `cold` recoveries spike without a planned restart or rebalance

## Remaining Gap

The current multinode hardening uses a shared durable store plus shared degraded-channel cache signaling when available.

That gives a credible clustered safety baseline, but it is not yet a full distributed consensus system for every partial partition scenario. If the cache layer is local-only instead of shared, degraded-channel propagation is node-local and operators should treat clustered recovery guarantees as weaker.
