# Push Benchmarking

Benchmark Sockudo capacity separately from provider delivery capacity. Large
tests must use fake tokens with provider dispatch disabled or pointed at a mock
provider. Real APNs, FCM, Web Push, HMS, and WNS tests should be small canaries
only.

## Benchmark Surfaces

| Surface | Command | Provider impact |
| --- | --- | --- |
| Internal pipeline | `make push-benchmark-internal` | None |
| HTTP admission/fanout | `make push-benchmark ARGS="..."` | None when provider dispatch is disabled or mocked |
| Scenario suite | `make push-benchmark-scenarios` | None by default; dry-run prints commands |
| Mock provider | `make push-mock-provider ARGS="--profile throttle"` | None |
| Real provider canary | `make push-provider-canary ARGS="--provider fcm --execute ..."` | Bounded by `--max-sends` and `--max-provider-attempts` |

## Scenario Suite

The suite is dry-run by default:

```bash
make push-benchmark-scenarios
```

Run one profile after confirming provider dispatch is disabled or mocked:

```bash
make push-benchmark-scenarios ARGS="--profile admission-baseline --provider-dispatch disabled --execute"
make push-benchmark-scenarios ARGS="--profile registry-fanout --provider-dispatch disabled --execute"
make push-benchmark-scenarios ARGS="--profile hot-channel --provider-dispatch disabled --execute"
make push-benchmark-scenarios ARGS="--profile all-provider-fanout --provider-dispatch disabled --execute"
make push-benchmark-scenarios ARGS="--profile quota-sweep --provider-dispatch disabled --execute"
make push-benchmark-scenarios ARGS="--profile soak --provider-dispatch disabled --execute"
```

The available profiles are:

| Profile | Purpose |
| --- | --- |
| `admission-baseline` | Signed publish acceptance latency and RPS. |
| `registry-fanout` | Device registration, channel subscription, fanout planning, queue/status pressure. |
| `hot-channel` | Single-channel fanout pressure. |
| `all-provider-admission` | Admission path with provider-specific registry shapes. |
| `all-provider-fanout` | Provider-specific fake-token registry and fanout planning. |
| `quota-sweep` | Acceptance behavior at increasing client concurrency. |
| `soak` | Longer-running queue/store stability check. |

## Direct Benchmarks

Admission only:

```bash
make push-benchmark ARGS="--mode admission --provider fcm --channels 100 --concurrency 100 --duration 120"
```

Registry and fanout:

```bash
make push-benchmark ARGS="--mode all --provider fcm --devices 100000 --channels 100 --concurrency 100 --duration 300"
```

Hot channel:

```bash
make push-benchmark ARGS="--mode all --provider fcm --devices 100000 --channels 1 --hot-channel --concurrency 50 --duration 120"
```

Provider-specific fake-token runs:

```bash
make push-benchmark ARGS="--mode all --provider apns --devices 10000 --channels 100 --concurrency 50 --duration 120"
make push-benchmark ARGS="--mode all --provider webpush --devices 10000 --channels 100 --concurrency 50 --duration 120"
make push-benchmark ARGS="--mode all --provider hms --devices 10000 --channels 100 --concurrency 50 --duration 120"
make push-benchmark ARGS="--mode all --provider wns --devices 10000 --channels 100 --concurrency 50 --duration 120"
```

## Mock Provider

Start a mock provider to test dispatch classification and backoff behavior
without touching external providers:

```bash
make push-mock-provider ARGS="--port 8791 --profile ok --latency-ms 25 --jitter-ms 10"
make push-mock-provider ARGS="--port 8791 --profile throttle --retry-after-seconds 2"
make push-mock-provider ARGS="--port 8791 --profile flaky --failure-rate 0.05"
make push-mock-provider ARGS="--port 8791 --profile invalid-token"
```

The mock exposes:

```bash
curl http://127.0.0.1:8791/health
curl http://127.0.0.1:8791/metrics
```

Provider dispatchers must be configured to use the mock provider base URL
before this protects external providers.

## Real Provider Canaries

Canaries are dry-run unless `--execute` is provided. They refuse to exceed
`--max-provider-attempts`, default `100`.

FCM:

```bash
make push-provider-canary ARGS="--provider fcm --token-file ./fcm_tokens.txt --max-sends 5 --upload-credential --service-account-json-file ./fcm_service_account.json --execute"
```

APNs:

```bash
make push-provider-canary ARGS="--provider apns --token-file ./apns_tokens.txt --max-sends 5 --upload-credential --pem-file ./apns.pem --execute"
```

Web Push:

```bash
make push-provider-canary ARGS="--provider webpush --recipient-json-file ./webpush_recipients.json --max-sends 5 --upload-credential --vapid-public-key '<public>' --vapid-private-key '<private>' --execute"
```

HMS:

```bash
make push-provider-canary ARGS="--provider hms --token-file ./hms_tokens.txt --max-sends 5 --upload-credential --hms-app-id '<app-id>' --hms-client-secret '<secret>' --execute"
```

WNS:

```bash
make push-provider-canary ARGS="--provider wns --token-file ./wns_channel_uris.txt --max-sends 5 --upload-credential --wns-package-sid '<sid>' --wns-client-secret '<secret>' --execute"
```

## Reporting

Report these separately:

- publish acceptance RPS and p50/p95/p99 latency
- registry seed throughput
- fanout planned/dispatched/failed counters
- queue lag per stage and provider
- Postgres write latency and connection saturation
- Iggy produce/consume/ack latency
- mock provider response profile and latency
- real provider canary volume, provider attempts, and observed throttling

Never turn a fake-token benchmark into a real-provider capacity claim.
