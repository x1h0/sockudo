## Overview

Sockudo V2 durable history lets you fetch stored messages for a single channel over HTTP without blocking the live publish path.

Guarantee level:

- Guaranteed: endpoint shape, cursor semantics, continuity metadata fields.
- Best-effort: retained span depends on configured history policy and backend health.
- Intentionally unsupported: direct browser/mobile access without a trusted backend proxy or signed backend caller.

History pages are:

- channel-scoped
- cursor-based, never offset-based
- ordered `newest_first` or `oldest_first`
- optionally bounded by serial, timestamp, or both

The public endpoint is:

`GET /apps/{appId}/channels/{channelName}/history`

## Query Model

Supported query parameters:

- `limit`
- `direction`
- `cursor`
- `start_serial`
- `end_serial`
- `start_time_ms`
- `end_time_ms`

`cursor` is opaque. Do not inspect or modify it. A cursor is only valid for:

- the same app
- the same channel
- the same direction
- the same bounds

If any of those change, Sockudo rejects the cursor with `400`.

## Continuity Metadata

Every history page includes a `continuity` object:

- `stream_id`
- `oldest_available_serial`
- `newest_available_serial`
- `oldest_available_published_at_ms`
- `newest_available_published_at_ms`
- `retained_messages`
- `retained_bytes`
- `complete`
- `truncated_by_retention`

This lets you distinguish:

- a partial page because more data is available: `has_more = true`
- a partial result because older data has already been evicted: `truncated_by_retention = true`

## Examples

Newest-first:

```bash
curl "http://127.0.0.1:6001/apps/app-1/channels/market:BTC/history?limit=50&direction=newest_first&auth_key=...&auth_timestamp=...&auth_version=1.0&auth_signature=..."
```

Bounded by serial:

```bash
curl "http://127.0.0.1:6001/apps/app-1/channels/market:BTC/history?start_serial=100&end_serial=150&direction=oldest_first&auth_key=...&auth_timestamp=...&auth_version=1.0&auth_signature=..."
```

Continue with a cursor:

```bash
curl "http://127.0.0.1:6001/apps/app-1/channels/market:BTC/history?cursor=eyJ2ZXJzaW9uIjoxLC4uLg&auth_key=...&auth_timestamp=...&auth_version=1.0&auth_signature=..."
```

## V1 And V2

This is a server-side API. V1 client behavior is unchanged.

Important:

- V1 clients still receive normal Pusher-compatible live deliveries.
- Durable history stores the canonical serialized V2 envelope used internally for continuity-aware history and future rewind/recovery work.
- Rewind and continuity-aware reconnect are separate features layered on top of this history substrate.

For browser/mobile apps, this endpoint should normally be accessed through a backend proxy or other trusted server-side component rather than directly from the client runtime.

## Presence History Is Separate

Presence history is not part of channel message history.

Use:

- `GET /apps/{appId}/channels/{channelName}/history` for retained channel messages
- `GET /apps/{appId}/channels/{channelName}/users` for the current presence member set
- `GET /apps/{appId}/channels/{channelName}/presence/history` for retained historical presence transitions on one presence channel

Important:

- `/users` returns current membership now.
- `/presence/history` returns retained joins/leaves and cleanup removals.
- On clustered deployments, duplicate join/remove reports collapse to one retained row per user transition until the opposite edge is recorded.
- `/presence/history` may report a degraded continuity state after a persistence failure even while live presence membership still works.
- Neither endpoint is a general-purpose database or cross-channel query surface.

## Errors

History requests return `400` for:

- invalid `direction`
- invalid bound combinations such as `start_serial > end_serial`
- malformed cursors
- expired cursors that point before the retained history floor
- cursors replayed against a different channel or different bounds
