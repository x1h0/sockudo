## Overview

Message annotations attach derived state to an existing versioned message. They are designed for reactions, read receipts, moderation votes, counters, and similar UI state where clients need the latest summary more than every intermediate change.

Annotations are a **Protocol V2 only** feature. They require:

- `versioned_messages.enabled = true`
- `annotations.enabled = true`
- `annotations_enabled = true` on the app channel policy or resolved namespace
- retained message state for the target channel

Enabling annotations for a channel forces message persistence for that channel. Plan retention and storage billing as if durable history is enabled for every annotated message.

## Annotation Types

Annotation types use:

```text
namespace:summarizer.version
```

Examples:

- `reactions:distinct.v1`
- `receipts:flag.v1`
- `moderation:multiple.v1`

The namespace is application-defined. The summarizer and version select Sockudo's projection behavior.

## Summarizers

### `total.v1`

Counts every create event and subtracts matching delete events. The same client can contribute more than once and unidentified annotations are allowed.

```json
{ "total": 12 }
```

Use for simple counters where contributor identity does not matter.

### `flag.v1`

Stores at most one active contribution per identified client. `client_id` is required.

```json
{
  "total": 2,
  "clientIds": ["user-1", "user-2"],
  "clipped": false
}
```

Use for read receipts, acknowledgements, or binary per-user state.

### `distinct.v1`

Tracks identified clients in named buckets. A client can appear in more than one name. `client_id` and `name` are required.

```json
{
  "like": {
    "total": 2,
    "clientIds": ["user-1", "user-2"],
    "clipped": false
  },
  "laugh": {
    "total": 1,
    "clientIds": ["user-1"],
    "clipped": false
  }
}
```

Use for reaction sets where one user may choose several reactions.

### `unique.v1`

Tracks identified clients in named buckets, but each client has only one active name. Re-publishing for the same client moves that client to the latest name. `client_id` and `name` are required.

```json
{
  "like": {
    "total": 1,
    "clientIds": ["user-2"],
    "clipped": false
  },
  "laugh": {
    "total": 1,
    "clientIds": ["user-1"],
    "clipped": false
  }
}
```

Use for one-choice reactions, ratings, or status selections.

### `multiple.v1`

Tracks per-client counts by name and also preserves anonymous counts in `totalUnidentified`.

```json
{
  "spam": {
    "total": 9,
    "clientCounts": {
      "moderator-1": 2,
      "moderator-2": 3
    },
    "totalUnidentified": 4,
    "clipped": false,
    "totalClientIds": 2
  }
}
```

Use for moderation scoring, weighted votes, or reports that may include anonymous submissions.

## Clipping

Summary payloads must fit the configured message payload limit. When contributor lists are too large, Sockudo rebuilds the projection with a smaller contributor list and sets `clipped: true`.

The total fields remain authoritative:

- `total` is still the complete count
- `clientIds` or `clientCounts` are partial
- `multiple.v1` also reports `totalClientIds`

Operators should monitor:

- `sockudo_annotation_summary_clipped_total{channel,type}`
- logs with `annotation summary clipped`

Frequent clipping means clients are depending on a contributor list that is too large for live payloads. Increase payload limits, reduce exposed contributor detail, or fetch full audit state through the raw annotation HTTP API.

## Channel Feature Flag

Enable globally and per channel or namespace:

```toml
[versioned_messages]
enabled = true

[annotations]
enabled = true

[[app_manager.array.apps.policy.channels.channel_namespaces]]
name = "chat"
channel_name_pattern = "^chat:[A-Za-z0-9._-]+$"
annotations_enabled = true
```

When `annotations_enabled` is false or omitted for the resolved channel, annotation publish attempts return `403`.

## Identified-Client Requirements

`flag.v1`, `distinct.v1`, and `unique.v1` require an identified client. The HTTP API can take `clientId` directly for trusted server calls or resolve it from `socketId` when the socket is signed in and has the right capability.

`multiple.v1` allows anonymous submissions. Anonymous counts appear in `totalUnidentified`.

## Subscribe Modes

Normal subscribers receive `message.summary` updates automatically. No special mode is required.

Raw annotation events are opt-in:

```json
{
  "modes": ["SUBSCRIBE", "ANNOTATION_SUBSCRIBE"]
}
```

Important: setting `modes` replaces default modes. Include `SUBSCRIBE` if the client should still receive normal channel messages.

## HTTP API

All endpoints use the normal signed `/apps/*` authentication model.

### Publish

```text
POST /apps/{appId}/channels/{channelName}/messages/{messageSerial}/annotations
```

Request:

```json
{
  "type": "reactions:distinct.v1",
  "name": "like",
  "clientId": "user-1",
  "count": 1,
  "data": { "source": "button" },
  "encoding": null
}
```

Response:

```json
{ "annotationSerial": "00000000000000000002:node:00000000000000000001" }
```

Requires `annotation-publish` when scoped to a signed-in socket.

### Delete

```text
DELETE /apps/{appId}/channels/{channelName}/messages/{messageSerial}/annotations/{annotationSerial}?socket_id={socketId}
```

Response:

```json
{
  "annotationSerial": "00000000000000000003:node:00000000000000000001",
  "deletedAnnotationSerial": "00000000000000000002:node:00000000000000000001"
}
```

Requires `annotation-delete-own` for the annotation owner or `annotation-delete-any` for moderation tools. Deleting a non-existent annotation returns `404`.

### List Raw Events

```text
GET /apps/{appId}/channels/{channelName}/messages/{messageSerial}/annotations?type=reactions:distinct.v1&from_serial=...&limit=100
```

Response:

```json
{
  "channel": "chat:room",
  "messageSerial": "msg:1",
  "limit": 100,
  "hasMore": false,
  "nextCursor": null,
  "items": []
}
```

Use this for audit and repair workflows. Normal clients should refresh summaries instead of replaying raw events for UX state.

## V2 Wire Events

Summary updates are ordinary channel deliveries on `sockudo_internal:message`:

```json
{
  "event": "sockudo_internal:message",
  "channel": "chat:room",
  "data": {
    "action": "message.summary",
    "serial": "msg:1",
    "annotations": {
      "summary": {
        "reactions:distinct.v1": {
          "like": {
            "total": 1,
            "clientIds": ["user-1"],
            "clipped": false
          }
        }
      }
    }
  },
  "extras": { "ephemeral": true }
}
```

Raw annotations are delivered only to subscribers with `ANNOTATION_SUBSCRIBE`:

```json
{
  "event": "sockudo_internal:annotation",
  "channel": "chat:room",
  "data": {
    "action": "annotation.create",
    "id": "annotation-id",
    "serial": "ann:1",
    "messageSerial": "msg:1",
    "type": "reactions:distinct.v1",
    "name": "like",
    "clientId": "user-1",
    "count": 1,
    "timestamp": 1775292000123
  }
}
```

## Operator Guidance

Monitor:

- `sockudo_annotations_published_total{channel,type}`
- `sockudo_annotations_deleted_total{channel,type}`
- `sockudo_annotation_summary_deliveries_total{channel}`
- `sockudo_annotation_summary_clipped_total{channel,type}`
- `sockudo_annotation_projection_rebuild_total{channel}`
- `sockudo_annotation_projection_rebuild_duration_seconds{channel}`

Investigate sustained clipping, repeated projection rebuilds on hot channels, and any error log containing `annotation event failed to fan out through cluster broadcast` or `annotation summary failed to fan out through cluster broadcast`.

Projection rebuilds are automatic on cache miss or stale projection. If rebuilds stay elevated, check backing-store latency and retention configuration before increasing traffic to the channel.

Retention implications:

- annotation summaries derive from retained annotation events
- annotated messages must remain queryable by message serial
- shortening retention can remove the source data needed to rebuild historical summaries
