The app manager is the source of truth for application credentials and app policy. Every WebSocket connection and API request is validated against the app manager.

## Drivers

| Driver | Best For | Persistence | External Dependency |
|---|---|---|---|
| `memory` | Development, single-node | None (lost on restart) | None |
| `mysql` | Production with MySQL stack | Full | MySQL 5.7+ |
| `postgres` | Production with PostgreSQL stack | Full | PostgreSQL 12+ |
| `dynamodb` | Serverless / AWS-native | Full | AWS DynamoDB |
| `surrealdb` | Document database deployments | Full | SurrealDB 3.x |
| `scylladb` | High-throughput, low-latency | Full | ScyllaDB / Cassandra |

Set the driver:

```bash
APP_MANAGER_DRIVER=memory
```

```json
{ "app_manager": { "driver": "memory" } }
```

## Memory Driver

The default driver. Apps are defined inline in `config/config.toml` (preferred) or `config/config.json`, or bootstrapped via environment variables. Data is not persisted and apps are lost on restart.

```toml
[app_manager]
driver = "memory"

[app_manager.array]
[[app_manager.array.apps]]
id = "my-app-id"
key = "my-app-key"
secret = "my-app-secret"
enabled = true
[app_manager.array.apps.policy.limits]
max_connections = 100000
max_client_events_per_second = 1000

[app_manager.array.apps.policy.features]
enable_client_messages = true

[app_manager.array.apps.policy.channels]
allowed_origins = ["*"]

[[app_manager.array.apps.policy.channels.channel_namespaces]]
name = "chat"
channel_name_pattern = "^chat:[A-Za-z0-9._-]+$"
max_channel_name_length = 200
allow_user_limited_channels = true
```

You can also bootstrap a default app entirely from environment variables:

```bash
SOCKUDO_DEFAULT_APP_ID=my-app-id
SOCKUDO_DEFAULT_APP_KEY=my-app-key
SOCKUDO_DEFAULT_APP_SECRET=my-app-secret
SOCKUDO_DEFAULT_APP_ENABLED=true
```

See [Environment Variables](/server/environment-variables#default-app-bootstrap) for all `SOCKUDO_DEFAULT_APP_*` options.

## MySQL Driver

Stores apps in a MySQL table with automatic schema migrations. Includes built-in query caching.

```json
{
  "app_manager": {
    "driver": "mysql"
  },
  "database": {
    "mysql": {
      "host": "localhost",
      "port": 3306,
      "username": "sockudo",
      "password": "secret",
      "database": "sockudo",
      "table_name": "applications"
    }
  }
}
```

### Connection Pooling

MySQL connections are pooled automatically:

| Setting | Default | Description |
|---|---|---|
| `connection_pool_size` | `10` | Pool size (legacy, used if pool_min/max not set) |
| `pool_min` | `null` | Minimum connections (overrides global) |
| `pool_max` | `null` | Maximum connections (overrides global) |

You can also set global pool limits:

```bash
DATABASE_POOLING_ENABLED=true
DATABASE_POOL_MIN=2
DATABASE_POOL_MAX=10
```

### Query Caching

App lookups are cached in-memory to avoid repeated database queries:

| Setting | Default | Description |
|---|---|---|
| `cache_ttl` | `300` | Cache TTL in seconds |
| `cache_cleanup_interval` | `60` | Cleanup interval in seconds |
| `cache_max_capacity` | `100` | Maximum cached entries |

### Table Schema

Sockudo auto-creates the table and migrates new columns automatically. Canonical app configuration is stored as a normalized `policy` document. Legacy scalar columns may still exist during compatibility migrations.

- `id`, `key`, `secret` - Core credentials
- `enabled` - App enabled flag
- `policy` - Canonical app policy document
- Legacy scalar columns may still be present for rollout compatibility

## PostgreSQL Driver

Same features as MySQL, adapted for PostgreSQL. Uses `JSONB` columns and PostgreSQL-native syntax.

```json
{
  "app_manager": {
    "driver": "postgres"
  },
  "database": {
    "postgres": {
      "host": "localhost",
      "port": 5432,
      "username": "sockudo",
      "password": "secret",
      "database": "sockudo"
    }
  }
}
```

```bash
APP_MANAGER_DRIVER=postgres
DATABASE_POSTGRES_HOST=localhost
DATABASE_POSTGRES_PORT=5432
DATABASE_POSTGRES_USERNAME=sockudo
DATABASE_POSTGRES_PASSWORD=secret
DATABASE_POSTGRES_DATABASE=sockudo
```

Connection pooling and query caching work identically to MySQL.

## DynamoDB Driver

Stores apps in a single DynamoDB table with the `id` field as the partition key.

```json
{
  "app_manager": {
    "driver": "dynamodb"
  },
  "database": {
    "dynamodb": {
      "region": "us-east-1",
      "table_name": "sockudo-applications"
    }
  }
}
```

For local development with [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html):

```bash
DATABASE_DYNAMODB_ENDPOINT_URL=http://localhost:8000
```

### Authentication

DynamoDB uses standard AWS credential resolution:

1. Environment variables: `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY`
2. AWS profile: `aws_profile_name` in config
3. IAM role (ECS task role, EC2 instance profile)

```bash
AWS_ACCESS_KEY_ID=AKIA...
AWS_SECRET_ACCESS_KEY=secret...
```

## SurrealDB Driver

Stores apps as SurrealDB documents keyed by app ID. Sockudo creates the table and a unique `key` index automatically at startup.

```json
{
  "app_manager": {
    "driver": "surrealdb"
  },
  "database": {
    "surrealdb": {
      "url": "ws://127.0.0.1:8000",
      "namespace": "sockudo",
      "database": "sockudo",
      "username": "root",
      "password": "root",
      "table_name": "applications"
    }
  }
}
```

```bash
APP_MANAGER_DRIVER=surrealdb
DATABASE_SURREALDB_URL=ws://127.0.0.1:8000
DATABASE_SURREALDB_NAMESPACE=sockudo
DATABASE_SURREALDB_DATABASE=sockudo
DATABASE_SURREALDB_USERNAME=root
DATABASE_SURREALDB_PASSWORD=root
DATABASE_SURREALDB_TABLE_NAME=applications
```

| Setting | Default | Description |
|---|---|---|
| `url` | `ws://127.0.0.1:8000` | SurrealDB endpoint URL |
| `namespace` | `sockudo` | Namespace selected at startup |
| `database` | `sockudo` | Database selected at startup |
| `username` | `root` | Username used for `signin()` |
| `password` | `root` | Password used for `signin()` |
| `table_name` | `applications` | Table storing app documents |
| `cache_ttl` | `300` | In-process lookup cache TTL in seconds |
| `cache_max_capacity` | `100` | Max cached app entries |

## ScyllaDB Driver

High-performance driver using ScyllaDB (or Apache Cassandra). Automatically creates the keyspace, table, and secondary indexes.

```json
{
  "app_manager": {
    "driver": "scylladb",
    "scylladb": {
      "nodes": ["10.0.0.1:9042", "10.0.0.2:9042"],
      "keyspace": "sockudo",
      "table_name": "applications",
      "username": "cassandra",
      "password": "cassandra",
      "replication_class": "NetworkTopologyStrategy",
      "replication_factor": 3
    }
  }
}
```

| Setting | Default | Description |
|---|---|---|
| `nodes` | `["127.0.0.1:9042"]` | Contact points |
| `keyspace` | `sockudo` | Keyspace name |
| `table_name` | `applications` | Table name |
| `replication_class` | `SimpleStrategy` | Replication strategy |
| `replication_factor` | `3` | Replication factor |

A secondary index on the `key` column is created automatically for fast key-based lookups.

## App Manager Caching

All database-backed drivers (MySQL, PostgreSQL, DynamoDB, SurrealDB, ScyllaDB) are automatically wrapped in a caching layer. This is separate from the [cache driver](/server/cache) and is configured under `app_manager.cache`:

```json
{
  "app_manager": {
    "cache": {
      "enabled": true,
      "ttl": 300
    }
  }
}
```

| Setting | Default | Description |
|---|---|---|
| `enabled` | `true` | Enable app lookup caching |
| `ttl` | `300` | Cache TTL in seconds |

The cache is:
- **Read-through**: on cache miss, fetches from the database and stores in cache.
- **Write-through**: on app update/delete, invalidates cache immediately.
- Keyed by both app ID and app key for fast lookups in either direction.

## App Configuration Fields

Each app supports these fields:

| Field | Type | Default | Description |
|---|---|---|---|
| `id` | `string` | required | Unique app identifier |
| `key` | `string` | required | Public app key (used in client connections) |
| `secret` | `string` | required | Secret for signing (never sent to clients) |
| `enabled` | `bool` | `true` | Accept connections for this app |
| `policy.features.enable_client_messages` | `bool` | `false` | Allow client-to-client events |
| `policy.limits.max_connections` | `u32` | `100` | Max simultaneous WebSocket connections |
| `policy.limits.max_client_events_per_second` | `u32` | `100` | Client event rate limit |
| `policy.limits.max_read_requests_per_second` | `u32?` | `100` | Read API rate limit |
| `policy.limits.max_backend_events_per_second` | `u32?` | `100` | Backend event rate limit |
| `policy.limits.max_presence_members_per_channel` | `u32?` | `100` | Max users per presence channel |
| `policy.limits.max_presence_member_size_in_kb` | `u32?` | `100` | Max presence user info size |
| `policy.limits.max_channel_name_length` | `u32?` | `100` | Max channel name length |
| `policy.limits.max_event_channels_at_once` | `u32?` | `100` | Max channels per broadcast |
| `policy.limits.max_event_name_length` | `u32?` | `100` | Max event name length |
| `policy.limits.max_event_payload_in_kb` | `u32?` | `100` | Max event payload size |
| `policy.limits.max_event_batch_size` | `u32?` | `100` | Max events per batch |
| `policy.features.enable_user_authentication` | `bool?` | `false` | Require user auth |
| `policy.features.enable_watchlist_events` | `bool?` | `false` | Enable watchlist events |
| `policy.channels.allowed_origins` | `string[]?` | `null` | Restrict WebSocket origins |
| `policy.webhooks` | `Webhook[]?` | `null` | Webhook definitions |
| `policy.channels.channel_delta_compression` | `object?` | `null` | Per-channel delta config overrides |

## Choosing a Driver

- **Development**: Use `memory` with inline apps or env vars.
- **Single-node production**: `mysql` or `postgres` with connection pooling.
- **Multi-node production**: Any database driver works since all nodes read from the same source.
- **AWS-native**: `dynamodb` with IAM roles.
- **Document database**: `surrealdb` when you want schemaless app documents with a unique key index.
- **High-throughput**: `scylladb` for lowest latency at scale.
