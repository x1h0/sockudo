## Overview

Idempotent publishing prevents duplicate message delivery when publishers retry HTTP requests. It is server-side deduplication: the publisher includes an `idempotency_key` with each publish request, and the server uses an atomic SET NX operation to ensure only the first request with that key triggers a broadcast. Retries with the same key receive a `200 OK` response but no duplicate broadcast is sent.

This is useful for:

- Unreliable networks where HTTP requests may be retried by clients or infrastructure
- At-least-once delivery systems (queues, webhooks) that may re-deliver publish requests
- Microservice architectures where multiple services might publish the same event
- Any publisher that cannot guarantee exactly-once HTTP delivery

## How It Differs from Connection Recovery

| | Connection Recovery | Idempotent Publishing |
|---|---|---|
| **Protects** | Subscribers (catch up on missed messages) | Publishers (prevent duplicate broadcasts) |
| **Mechanism** | Serial-based replay buffer | Atomic SET NX key deduplication |
| **Protocol** | V2 only (WebSocket) | V1 and V2 (REST API) |
| **Scope** | Per-channel message replay | Per-request deduplication |

These two features complement each other. Connection recovery ensures subscribers do not miss messages during brief disconnects. Idempotent publishing ensures publishers do not cause duplicate broadcasts from retries. Together they provide exactly-once semantics end-to-end.

## Configuration

### TOML Config

```toml [config/config.toml]
[idempotency]
enabled = true
ttl_seconds = 120
max_key_length = 128
```

| Field | Type | Default | Description |
|---|---|---|---|
| `enabled` | `bool` | `true` | Enable idempotent publishing |
| `ttl_seconds` | `u64` | `120` | How long to remember an idempotency key (seconds) |
| `max_key_length` | `usize` | `128` | Maximum length of an idempotency key string |

### Per-App Override

Override idempotency settings for a specific app in your app configuration:

```toml [config/config.toml]
[[app_manager.array.apps]]
id = "app-id"
key = "app-key"
secret = "app-secret"

[app_manager.array.apps.policy.idempotency]
enabled = true
ttl_seconds = 300
max_key_length = 256
```

### Environment Variables

```bash
IDEMPOTENCY_ENABLED=true
IDEMPOTENCY_TTL_SECONDS=120
IDEMPOTENCY_MAX_KEY_LENGTH=128
```

### Cargo Feature Flag

Idempotent publishing is always compiled in. It is part of the HTTP publish path, not the WebSocket protocol, so it works for both V1 and V2 connections. There is no separate feature flag to enable at compile time.

## How to Use

### Single Event

Include an `idempotency_key` in the publish request body:

```bash
curl -X POST http://localhost:6001/apps/app-id/events \
  -H "Content-Type: application/json" \
  -d '{
    "name": "order-placed",
    "channel": "orders",
    "data": "{\"order_id\":\"abc-123\",\"total\":99.95}",
    "idempotency_key": "order-abc-123-v1"
  }'
```

Alternatively, set the `X-Idempotency-Key` header:

```bash
curl -X POST http://localhost:6001/apps/app-id/events \
  -H "Content-Type: application/json" \
  -H "X-Idempotency-Key: order-abc-123-v1" \
  -d '{
    "name": "order-placed",
    "channel": "orders",
    "data": "{\"order_id\":\"abc-123\",\"total\":99.95}"
  }'
```

If both are present, the body field takes precedence over the header.

### Batch Events

For batch publishes, each event in the batch can have its own `idempotency_key`. The batch request can also have an `X-Idempotency-Key` header that applies to the entire batch:

```bash
curl -X POST http://localhost:6001/apps/app-id/batch_events \
  -H "Content-Type: application/json" \
  -H "X-Idempotency-Key: batch-20260324-001" \
  -d '{
    "batch": [
      {
        "name": "order-placed",
        "channel": "orders",
        "data": "{\"order_id\":\"abc-123\"}",
        "idempotency_key": "order-abc-123"
      },
      {
        "name": "order-placed",
        "channel": "orders",
        "data": "{\"order_id\":\"def-456\"}",
        "idempotency_key": "order-def-456"
      }
    ]
  }'
```

Per-event keys deduplicate individual events within the batch. The batch-level header key deduplicates the entire batch request.

### Key Requirements

- The key can be any string, typically a UUID or a domain-specific identifier (e.g., `order-{id}-v{version}`)
- Maximum length is configurable via `max_key_length` (default 128 characters)
- Keys are scoped per app -- the same key in different apps is treated independently
- Keys expire after `ttl_seconds` -- after expiry, the same key will trigger a new broadcast

### Retry Behavior

| Scenario | Result |
|---|---|
| First request with a key | Event is broadcast, key is stored, `200 OK` returned |
| Retry with the same key (within TTL) | No broadcast, cached `200 OK` returned |
| Same key after TTL expires | Treated as a new request, event is broadcast again |
| Request without any key | Event is always broadcast (no deduplication) |

## Prometheus Metrics

When idempotent publishing is enabled, two metrics are exposed at the `/metrics` endpoint:

```bash
curl http://localhost:9601/metrics | grep idempotency
```

| Metric | Type | Description |
|---|---|---|
| `sockudo_idempotency_publish_total` | Counter | Total publish requests that included an idempotency key |
| `sockudo_idempotency_duplicates_total` | Counter | Publish requests deduplicated (key already seen within TTL) |

A high `sockudo_idempotency_duplicates_total` relative to `sockudo_idempotency_publish_total` indicates frequent retries from publishers. This is expected in at-least-once delivery systems but may warrant investigation if the ratio is unexpectedly high.

## Message-Level Idempotency via Extras (V2 Only)

In addition to HTTP-level idempotency (which deduplicates the HTTP request itself), V2 connections support message-level idempotency through the `extras.idempotency_key` field on broadcast messages. These are two distinct mechanisms:

| | HTTP Idempotency | Message-Level Idempotency |
|---|---|---|
| **Scope** | Deduplicates the HTTP publish request | Deduplicates at the broadcast level per channel |
| **Protocol** | V1 and V2 (REST API) | V2 only (WebSocket `extras` field) |
| **Key location** | `idempotency_key` in request body or `X-Idempotency-Key` header | `extras.idempotency_key` on the message payload |
| **Dedup scope** | Per app + key | Per app + channel + key |

When a V2 message includes `extras.idempotency_key`, the server stores the key in the cache with the format:

```
idempotency:{app_id}:{channel}:{key}
```

The same key on different channels results in independent publishes -- each channel is treated separately. The TTL and max key length are governed by the `[idempotency]` server config section.

### Example V2 Message with Idempotency Key

```json
{
  "event": "order-placed",
  "channel": "orders",
  "data": "{\"order_id\":\"abc-123\"}",
  "extras": {
    "idempotency_key": "order-abc-123-v1"
  }
}
```

If the same message is published again with the same `extras.idempotency_key` to the same channel, it will be silently dropped. Publishing to a different channel with the same key is an independent operation.
