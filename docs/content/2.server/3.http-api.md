## Endpoints

### Realtime Publish

- `POST /apps/{appId}/events` — publish a single event to one or more channels
- `POST /apps/{appId}/batch_events` — publish multiple events in one request

### Channel Queries

- `GET /apps/{appId}/channels` — list occupied channels, with optional prefix filter
- `GET /apps/{appId}/channels/{channelName}` — get info for a specific channel
- `GET /apps/{appId}/channels/{channelName}/history` — fetch durable history for a single channel
- `GET /apps/{appId}/channels/{channelName}/presence/history` — fetch retained presence event history for a single presence channel
- `GET /apps/{appId}/channels/{channelName}/presence/history/state` — inspect the authoritative presence-history stream state for a single presence channel
- `POST /apps/{appId}/channels/{channelName}/presence/history/reset` — rotate and clear a presence-history stream after unrecoverable continuity loss
- `GET /apps/{appId}/channels/{channelName}/history/state` — inspect the authoritative durable stream state for a single channel
- `POST /apps/{appId}/channels/{channelName}/history/reset` — intentionally rotate and purge a channel stream
- `POST /apps/{appId}/channels/{channelName}/history/purge` — explicitly purge retained history without rotating the stream
- `GET /apps/{appId}/channels/{channelName}/users` — list members of a presence channel

### User Management

- `POST /apps/{appId}/users/{userId}/terminate_connections` — disconnect all sockets for a user

### Health and Ops

- `GET /up` — server health check
- `GET /up/{appId}` — per-app health check
- `GET /usage` — memory usage (can be disabled via `http_api.usage_enabled: false`)
- `GET /stats` — server and per-app connection statistics (same gate as `/usage`)
- `GET /metrics` — Prometheus metrics (served on the metrics port, default `9601`)

## Publish Single Event

`POST /apps/{appId}/events`

Broadcasts an event to one or more channels.

**Request body**

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | Yes | Event name (e.g. `"my-event"`) |
| `data` | string | Yes | Event payload. Must be a JSON-encoded string; max size enforced by `max_event_payload_in_kb` |
| `channel` | string | One of | Target channel name. Use either `channel` or `channels`, not both |
| `channels` | string[] | One of | List of target channels (max 100 by default) |
| `socket_id` | string | No | Socket to exclude from delivery |
| `info` | string | No | Comma-separated list of info fields to return: `subscription_count`, `user_count`, `cache` |
| `tags` | object | No | Key-value tags for server-side tag filtering (V2 connections only) |
| `idempotency_key` | string | No | Deduplication key; duplicate publishes with the same key within the TTL window are ignored |

**Response** `200 OK`

If `info` was not requested:

```json
{}
```

If `info` was requested, the response contains per-channel stats:

```json
{
  "channels": {
    "my-channel": {
      "subscription_count": 10,
      "user_count": 5
    }
  }
}
```

`user_count` is only populated for presence channels. `cache` returns the last cached payload for cache channels.

**Error responses**

| Status | Reason |
|---|---|
| `400` | Missing required field, invalid JSON, `channel` and `channels` both set, too many channels, or `user_count` requested on a non-presence channel |
| `401` | Invalid or missing auth signature |
| `413` | Event payload exceeds the configured size limit |

## Publish Batch Events

`POST /apps/{appId}/batch_events`

Publishes multiple events in a single request. Each item in the `batch` array follows the same shape as a single event publish. The max batch size is configurable (`max_batch_messages`, default 10).

**Request body**

```json
{
  "batch": [
    {
      "name": "event-one",
      "channel": "ch-a",
      "data": "{\"hello\":\"world\"}",
      "info": "subscription_count",
      "idempotency_key": "key-1"
    },
    {
      "name": "event-two",
      "channels": ["ch-b", "ch-c"],
      "data": "{\"foo\":\"bar\"}",
      "idempotency_key": "key-2"
    }
  ]
}
```

**Response** `200 OK`

Returns a `batch` array in the same order. Each entry is an object with the requested `info` fields, or an empty object if no info was requested.

```json
{
  "batch": [
    { "subscription_count": 5 },
    {}
  ]
}
```

**Error responses** — same codes as the single-event endpoint; a single invalid item rejects the entire request.

## List Channels

`GET /apps/{appId}/channels`

Returns all occupied channels for the app.

**Query parameters**

| Parameter | Description |
|---|---|
| `filter_by_prefix` | Only return channels whose name starts with this value (e.g. `presence-`) |
| `info` | Comma-separated info fields: `subscription_count`, `user_count` |

`user_count` is only available when `filter_by_prefix=presence-`.

**Response** `200 OK`

```json
{
  "channels": {
    "my-channel": {
      "subscription_count": 4
    },
    "presence-room": {
      "subscription_count": 3,
      "user_count": 3
    }
  }
}
```

Channels with no subscribers are not included.

## Get Channel Info

`GET /apps/{appId}/channels/{channelName}`

**Query parameters**

| Parameter | Description |
|---|---|
| `info` | Comma-separated: `subscription_count`, `user_count`, `cache` |

`user_count` is only valid for presence channels (name must start with `presence-`). Requesting it on any other channel type returns `400`.

**Response** `200 OK`

```json
{
  "occupied": true,
  "subscription_count": 10,
  "user_count": 5,
  "cache": "{\"last\":\"payload\"}"
}
```

`cache` is only populated for cache channels and only when explicitly requested via `info=cache`.

## Get Presence Channel Users

`GET /apps/{appId}/channels/{channelName}/users`

Only works on presence channels (name must start with `presence-`).

**Response** `200 OK`

```json
{
  "users": [
    { "id": "user-1" },
    { "id": "user-2" }
  ]
}
```

Returns `400` if the channel is not a presence channel.

Important:

- This endpoint returns the current member set.
- It does not return historical joins/leaves.
- Use `GET /apps/{appId}/channels/{channelName}/presence/history` for retained presence events.

## Get Presence History

`GET /apps/{appId}/channels/{channelName}/presence/history`

Returns retained presence event history for a single presence channel using opaque cursor pagination.

This is event history, not current membership state.

Use it to answer:

- who joined or left during a bounded period
- what presence transitions were retained for this one channel

Do not use it as a general-purpose presence database.

**Query parameters**

| Parameter | Description |
|---|---|
| `limit` | Page size. Defaults to `presence_history.max_page_size` and is always capped server-side |
| `direction` | `newest_first` or `oldest_first` |
| `cursor` | Opaque cursor returned by a previous presence-history response |
| `start_serial` | Optional inclusive lower serial bound |
| `end_serial` | Optional inclusive upper serial bound |
| `start_time_ms` | Optional inclusive lower timestamp bound (Unix milliseconds) |
| `end_time_ms` | Optional inclusive upper timestamp bound (Unix milliseconds) |

This endpoint only works on presence channels. Invalid, mismatched, and expired cursors return `400`.

**Response** `200 OK`

```json
{
  "items": [
    {
      "stream_id": "c5c6f9d2-12a7-4c6d-9b0a-0f8f7c4b77a4",
      "serial": 14,
      "published_at_ms": 1775292000456,
      "event": "member_removed",
      "cause": "disconnect",
      "user_id": "user-42",
      "connection_id": "230423.3434",
      "dead_node_id": null,
      "payload_size_bytes": 188,
      "presence_event": {
        "stream_id": "c5c6f9d2-12a7-4c6d-9b0a-0f8f7c4b77a4",
        "serial": 14,
        "published_at_ms": 1775292000456,
        "event": "member_removed",
        "cause": "disconnect",
        "user_id": "user-42",
        "connection_id": "230423.3434",
        "user_info": {
          "name": "Ada"
        },
        "dead_node_id": null
      }
    }
  ],
  "direction": "newest_first",
  "limit": 50,
  "has_more": false,
  "next_cursor": null,
  "bounds": {
    "start_serial": null,
    "end_serial": null,
    "start_time_ms": null,
    "end_time_ms": null
  },
  "continuity": {
    "stream_id": "c5c6f9d2-12a7-4c6d-9b0a-0f8f7c4b77a4",
    "oldest_available_serial": 1,
    "newest_available_serial": 14,
    "oldest_available_published_at_ms": 1775291000000,
    "newest_available_published_at_ms": 1775292000456,
    "retained_events": 14,
    "retained_bytes": 2630,
    "degraded": false,
    "complete": true,
    "truncated_by_retention": false
  },
  "stream_state": {
    "stream_id": "c5c6f9d2-12a7-4c6d-9b0a-0f8f7c4b77a4",
    "durable_state": "healthy",
    "continuity_proven": true,
    "reset_required": false,
    "reason": null,
    "node_id": null,
    "last_transition_at_ms": null,
    "authoritative_source": "in_memory",
    "observed_source": "in_memory"
  }
}
```

**Examples**

Newest-first:

```bash
curl "http://127.0.0.1:6001/apps/app-1/channels/presence-chat/presence/history?limit=50&direction=newest_first&auth_key=...&auth_timestamp=...&auth_version=1.0&auth_signature=..."
```

Bounded by serial:

```bash
curl "http://127.0.0.1:6001/apps/app-1/channels/presence-chat/presence/history?start_serial=10&end_serial=25&direction=oldest_first&auth_key=...&auth_timestamp=...&auth_version=1.0&auth_signature=..."
```

Bounded by time:

```bash
curl "http://127.0.0.1:6001/apps/app-1/channels/presence-chat/presence/history?start_time_ms=1775291000000&end_time_ms=1775292000000&direction=oldest_first&auth_key=...&auth_timestamp=...&auth_version=1.0&auth_signature=..."
```

Continue with a cursor:

```bash
curl "http://127.0.0.1:6001/apps/app-1/channels/presence-chat/presence/history?cursor=eyJ2ZXJzaW9uIjoxLC4uLg&auth_key=...&auth_timestamp=...&auth_version=1.0&auth_signature=..."
```

`continuity.complete` is only `true` when Sockudo can fully satisfy the requested slice and the channel is not marked degraded. `continuity.degraded = true` means live presence may still be functioning, but retained history should not be treated as complete after a persistence failure. `continuity.truncated_by_retention = true` means older retained transitions have already expired.

`stream_state` explains why continuity is or is not trusted:

- `durable_state = healthy`: no known persistence incident for the current stream
- `durable_state = degraded`: a write failed, but the stream had no retained floor to repair yet
- `durable_state = reset_required`: a write failed after retained events already existed, so continuity for the current stream can no longer be proven without an explicit reset

**Error responses**

| Status | Reason |
|---|---|
| `400` | Non-presence channel, invalid direction, invalid bounds such as `start_serial > end_serial`, malformed cursor, expired cursor, or cursor replayed against different bounds |
| `401` | Invalid or missing auth signature |

`GET /apps/{appId}/channels/{channelName}/history`

Returns durable message history for a single channel using opaque cursor pagination.

**Query parameters**

| Parameter | Description |
|---|---|
| `limit` | Page size. Defaults to `history.max_page_size` and is always capped server-side |
| `direction` | `newest_first` or `oldest_first` |
| `cursor` | Opaque cursor returned by a previous history response |
| `start_serial` | Optional inclusive lower serial bound |

| `start_time_ms` | Optional inclusive lower timestamp bound (Unix milliseconds) |
| `end_time_ms` | Optional inclusive upper timestamp bound (Unix milliseconds) |

```json
    "authoritative_source": "durable_store",
    "observed_source": "durable_store"
  }

```

`has_more` describes pagination state. `continuity.complete` is only `true` when the requested slice is fully satisfied and no additional page is required. `continuity.truncated_by_retention` becomes `true` when the requested lower bound extends beyond the retained history floor.

Paginated usage pattern:

```bash [Terminal]
GET /apps/{appId}/channels/market:BTC/history?limit=50&direction=newest_first
GET /apps/{appId}/channels/market:BTC/history?cursor=<next_cursor_from_previous_page>
```

**Compatibility**

- The HTTP history endpoint is a server-side API. It does not change WebSocket behavior for V1 clients.
- History is stored from the canonical V2 message envelope, so responses include V2-style metadata such as `serial` and the durable `stream_id`.
- V1 publishers and subscribers remain supported; the endpoint simply does not change their live delivery format.

**Guarantee level**

- Guaranteed: request/response shape, cursor validation rules, continuity metadata fields.
- Best-effort: retained depth depends on configured history policy and backend health.
- Intentionally unsupported: direct browser/mobile use without a trusted backend proxy or signed server-side caller.

### `POST /apps/{appId}/channels/{channelName}/history/reset`

Destructive operator control that rotates `stream_id` and purges all retained history for the channel.

This endpoint is intentionally hard to call by accident:

- it requires the same signed server-side API authentication as other app management endpoints
- `confirm_channel` must exactly match the path channel
- `confirm_operation` must be `"reset"`
- `reason` must be non-empty

**Request body**

```json
{
  "confirm_channel": "market:BTC",
  "confirm_operation": "reset",
  "reason": "irrecoverable continuity loss after storage incident",
  "requested_by": "ops@example.com"
}
```

**Response** `200 OK`

```json
{
  "ok": true,
  "operation": "reset",
  "channel": "market:BTC",
  "reason": "irrecoverable continuity loss after storage incident",
  "requested_by": "ops@example.com",
  "previous_stream_id": "old-stream",
  "new_stream_id": "new-stream",
  "purged_messages": 57,
  "purged_bytes": 9128,
  "stream": {
    "stream_id": "new-stream",
    "next_serial": 1,
    "retained": {
      "retained_messages": 0,
      "retained_bytes": 0,
      "oldest_available_serial": null,
      "newest_available_serial": null
    },
    "state": {
      "durable_state": "healthy",
      "recovery_allowed": true,
      "reset_required": false
    }
  }
}
```

Client impact:

- reconnect and resume attempts that present the old `stream_id` will fail with `stream_reset`
- rewind and history requests start from the new empty stream
- this is the right control when the old stream must no longer be considered continuous

### `POST /apps/{appId}/channels/{channelName}/history/purge`

Destructive operator control that deletes retained history rows without rotating `stream_id`.

Supported modes:

- `all`
- `before_serial`
- `before_time_ms`

**Request body**

```json
{
  "confirm_channel": "market:BTC",
  "confirm_operation": "purge",
  "mode": "before_serial",
  "before_serial": 4000,
  "reason": "trim backlog after operator review",
  "requested_by": "ops@example.com"
}
```

`before_serial` and `before_time_ms` are exclusive lower-bound trims. For example, `before_serial: 4000` deletes rows with serials `< 4000` and preserves serial `4000` onward.

Client impact:

- the stream stays on the same `stream_id`
- clients behind the new retained floor fail with `position_expired`, not `stream_reset`
- use reset instead of purge when you need to break continuity intentionally

## Health Checks

### `GET /up`

General server health. No authentication required.

Checks the adapter, cache manager, and queue/webhook system. Non-critical subsystems (queue/webhooks) cause a `degraded` state rather than an error.

| Status | HTTP code | `X-Health-Check` header |
|---|---|---|
| All systems healthy | `200` | `ok` |
| Non-critical system degraded | `304` | `degraded` |
| Critical system unhealthy | `503` | `error` |

### `GET /up/{appId}`

Same checks as `/up`, plus validates that the app exists and is enabled. Returns `404` if the app is not found or is disabled.

## Signed Query Parameters

All routes under `/apps/*` require Pusher-style query auth:

- `auth_key` — app key
- `auth_timestamp` — Unix timestamp (seconds)
- `auth_version` — always `"1.0"`
- `auth_signature` — HMAC-SHA256 hex digest
- `body_md5` — MD5 hex digest of the request body (required for non-empty `POST` bodies)

The timestamp must be within ±10 minutes of the server clock.

## Signature Format

String to sign:

```text
{METHOD}\n{REQUEST_PATH}\n{SORTED_QUERY_EXCLUDING_AUTH_SIGNATURE}
```

Query params are sorted alphabetically by key and joined with `&`. `auth_signature` itself is excluded from the string before signing.

## Signing Example

::code-group

```ts [Node.js]
import crypto from "node:crypto";

export function signPusherRequest({
  method,
  path,
  appKey,
  appSecret,
  body,
}: {
  method: "GET" | "POST";
  path: string;
  appKey: string;
  appSecret: string;
  body?: string;
}) {
  const auth_timestamp = Math.floor(Date.now() / 1000).toString();
  const params: Record<string, string> = {
    auth_key: appKey,
    auth_timestamp,
    auth_version: "1.0",
  };

  if (method === "POST" && body && body.length > 0) {
    params.body_md5 = crypto.createHash("md5").update(body).digest("hex");
  }

  const queryForSig = Object.keys(params)
    .sort()
    .map((k) => `${k}=${params[k]}`)
    .join("&");

  const stringToSign = `${method}\n${path}\n${queryForSig}`;
  const auth_signature = crypto
    .createHmac("sha256", appSecret)
    .update(stringToSign)
    .digest("hex");

  return { ...params, auth_signature };
}
```

```go [Go]
package main

import (
    "crypto/hmac"
    "crypto/md5"
    "crypto/sha256"
    "encoding/hex"
    "fmt"
    "net/url"
    "sort"
    "strconv"
    "strings"
    "time"
)

func signRequest(method, path, appKey, appSecret, body string) url.Values {
    params := url.Values{}
    params.Set("auth_key", appKey)
    params.Set("auth_timestamp", strconv.FormatInt(time.Now().Unix(), 10))
    params.Set("auth_version", "1.0")

    if method == "POST" && body != "" {
        h := md5.Sum([]byte(body))
        params.Set("body_md5", hex.EncodeToString(h[:]))
    }

    keys := make([]string, 0, len(params))
    for k := range params {
        keys = append(keys, k)
    }
    sort.Strings(keys)
    parts := make([]string, 0, len(keys))
    for _, k := range keys {
        parts = append(parts, k+"="+params.Get(k))
    }
    queryForSig := strings.Join(parts, "&")

    stringToSign := fmt.Sprintf("%s\n%s\n%s", method, path, queryForSig)
    mac := hmac.New(sha256.New, []byte(appSecret))
    mac.Write([]byte(stringToSign))
    params.Set("auth_signature", hex.EncodeToString(mac.Sum(nil)))
    return params
}
```

```php [PHP]
function signRequest(string $method, string $path, string $appKey, string $appSecret, string $body = ''): array {
    $params = [
        'auth_key'       => $appKey,
        'auth_timestamp' => time(),
        'auth_version'   => '1.0',
    ];

    if ($method === 'POST' && $body !== '') {
        $params['body_md5'] = md5($body);
    }

    ksort($params);
    $queryForSig = http_build_query($params);
    $stringToSign = implode("\n", [$method, $path, $queryForSig]);
    $params['auth_signature'] = hash_hmac('sha256', $stringToSign, $appSecret);

    return $params;
}
```

```python [Python]
import hashlib
import hmac
import time
from urllib.parse import urlencode

def sign_request(method: str, path: str, app_key: str, app_secret: str, body: str = "") -> dict:
    params = {
        "auth_key": app_key,
        "auth_timestamp": str(int(time.time())),
        "auth_version": "1.0",
    }

    if method == "POST" and body:
        params["body_md5"] = hashlib.md5(body.encode()).hexdigest()

    query_for_sig = urlencode(sorted(params.items()))
    string_to_sign = f"{method}\n{path}\n{query_for_sig}"
    signature = hmac.new(app_secret.encode(), string_to_sign.encode(), hashlib.sha256).hexdigest()
    params["auth_signature"] = signature
    return params
```

```ruby [Ruby]
require 'openssl'
require 'digest'

def sign_request(method, path, app_key, app_secret, body = '')
  params = {
    'auth_key'       => app_key,
    'auth_timestamp' => Time.now.to_i.to_s,
    'auth_version'   => '1.0',
  }

  params['body_md5'] = Digest::MD5.hexdigest(body) if method == 'POST' && !body.empty?

  query_for_sig = params.sort.map { |k, v| "#{k}=#{v}" }.join('&')
  string_to_sign = "#{method}\n#{path}\n#{query_for_sig}"
  auth_signature = OpenSSL::HMAC.hexdigest('sha256', app_secret, string_to_sign)

  params.merge('auth_signature' => auth_signature)
end
```

## Publish Example

::code-group

```ts [Node.js]
import { signPusherRequest } from "./sign";

const path = "/apps/app-id/events";
const body = JSON.stringify({
  name: "my-event",
  channel: "my-channel",
  data: JSON.stringify({ hello: "world" }),
});

const auth = signPusherRequest({
  method: "POST",
  path,
  appKey: "app-key",
  appSecret: "app-secret",
  body,
});

const qs = new URLSearchParams(auth).toString();
await fetch(`http://127.0.0.1:6001${path}?${qs}`, {
  method: "POST",
  headers: { "content-type": "application/json" },
  body,
});
```

## Common Query Patterns

```text
# List only presence channels with member counts
GET /apps/{appId}/channels?filter_by_prefix=presence-&info=user_count

# Get subscription and member count for one channel
GET /apps/{appId}/channels/{name}?info=subscription_count,user_count

# Get cached payload for a cache channel
GET /apps/{appId}/channels/{name}?info=cache

# Publish and get back subscriber count
POST /apps/{appId}/events?...&info=subscription_count
```
