## Overview

Ephemeral messages are fire-and-forget events that bypass the recovery buffer and webhook pipeline. They are delivered to currently connected subscribers but are never stored for replay or forwarded to webhook endpoints. This makes them ideal for high-frequency, transient data where missing a message during a brief disconnect is acceptable.

::callout{type="info"}
Ephemeral messages are a **protocol V2 only** feature. V1 connections still receive ephemeral messages (with extras stripped), but the ephemeral flag itself only has effect when published via the V2 extras envelope.
::

## Configuration

Ephemeral handling can be disabled globally:

```toml [config/config.toml]
[ephemeral]
enabled = true
```

Or via environment variable:

```bash
EPHEMERAL_ENABLED=true
```

When disabled, `extras.ephemeral` is ignored and messages are treated as normal persistent publishes.

## Use Cases

- **Typing indicators** -- transient UI state that becomes stale in milliseconds
- **Live cursors** -- collaborative editing cursor positions
- **GPS / location updates** -- high-frequency position data where only the latest matters
- **IoT telemetry** -- sensor readings streamed at sub-second intervals
- **Game state ticks** -- frame-by-frame state that is superseded every tick

## What Ephemeral Bypasses

| Component | Bypassed? |
|---|---|
| Recovery buffer | Yes -- message is not stored for replay |
| Webhooks | Yes -- no `channel_occupied` / message webhooks fired |
| Serial number assignment | No -- V2 connections still receive a `serial` |
| Delta compression | No -- still applied if enabled |
| Tag filtering | No -- still evaluated |
| Echo control | No -- still respected |
| Idempotency deduplication | No -- still deduplicated if `idempotency_key` is set |
| Rate limiting | No -- still enforced |

The key distinction: ephemeral controls **persistence and side-effects**, not delivery filtering.

## Publishing an Ephemeral Message

Set `extras.ephemeral` to `true` in the publish body:

```json
POST /apps/{app_id}/events
{
  "name": "cursor-move",
  "channel": "collaboration-room",
  "data": "{\"user\":\"alice\",\"x\":450,\"y\":230}",
  "extras": {
    "ephemeral": true
  }
}
```

### Client Events

V2 client SDKs can trigger ephemeral client events:

```javascript
channel.trigger('client-cursor', { x: 450, y: 230 }, {
  extras: { ephemeral: true }
});
```

### Batch Publishing

Mix ephemeral and persistent events in a single batch:

```json
POST /apps/{app_id}/batch_events
{
  "batch": [
    {
      "name": "cursor-move",
      "channel": "room-1",
      "data": "{\"x\":100,\"y\":200}",
      "extras": { "ephemeral": true }
    },
    {
      "name": "document-change",
      "channel": "room-1",
      "data": "{\"op\":\"insert\",\"pos\":42,\"text\":\"hello\"}"
    }
  ]
}
```

The cursor event skips the buffer; the document change is buffered for recovery.

## Cluster Behavior

In a multi-node deployment with Redis or NATS adapters, ephemeral messages are propagated to all nodes in the cluster. Remote nodes also skip buffering and webhooks for the message. The ephemeral flag travels with the inter-node broadcast so behavior is consistent regardless of which node the subscriber is connected to.

## Monitoring

Sockudo exposes a Prometheus counter for ephemeral messages:

```
sockudo_ephemeral_messages_total
```

This counter increments for every ephemeral message published. Use it to track the ratio of ephemeral to persistent traffic:

```promql
rate(sockudo_ephemeral_messages_total[5m])
  / rate(sockudo_messages_total[5m])
```

## When NOT to Use Ephemeral

Avoid ephemeral for events where missed messages cause data loss:

- **Chat messages** -- users expect to see messages they missed during a reconnect
- **Order updates** -- state transitions must not be lost
- **Configuration changes** -- all subscribers must eventually receive the update

For these cases, use standard (non-ephemeral) publishing with [connection recovery](/server/connection-recovery) enabled.
