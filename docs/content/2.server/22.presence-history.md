## Overview

Sockudo records presence transitions (member joins and leaves) as a durable, queryable history stream per presence channel. This is separate from:

- **Live presence** (`/channels/{channel}/users`) — current members right now
- **Channel message history** (`/channels/{channel}/history`) — retained published messages

Presence history answers: "who joined and left this channel, and when?"

Guarantee level:

- Guaranteed: endpoint shapes, cursor semantics, continuity metadata fields, deduplication.
- Best-effort: retained span depends on configured retention policy and backend health.
- Intentionally unsupported: direct browser/mobile access. Client SDKs must use a backend proxy.

## Enabling Presence History

Presence history is **disabled by default**. Enable it in config or environment:

::code-group
```toml [config.toml]
[presence_history]
enabled = true
retention_window_seconds = 86400   # 24 hours
max_page_size = 100
# max_events_per_channel = 10000   # optional cap
# max_bytes_per_channel = 10485760 # optional 10 MB cap
```

```bash [Environment]
PRESENCE_HISTORY_ENABLED=true
PRESENCE_HISTORY_RETENTION_WINDOW_SECONDS=86400
PRESENCE_HISTORY_MAX_PAGE_SIZE=100
```
::

Per-app and per-namespace overrides are available. Resolution order: namespace > app > global.

## Endpoints

All endpoints require HMAC-signed requests (same auth as other REST APIs).

### Read History

```
GET /apps/{appId}/channels/{channelName}/presence/history
```

Query parameters:

| Parameter | Type | Description |
|---|---|---|
| `limit` | integer | Max items per page (capped by `max_page_size`) |
| `direction` | string | `newest_first` (default) or `oldest_first`. Aliases: `backwards`/`reverse`, `forwards`/`forward` |
| `cursor` | string | Opaque pagination cursor from a previous page |
| `start_serial` | integer | Only return events at or after this serial |
| `end_serial` | integer | Only return events at or before this serial |
| `start_time_ms` | integer | Only return events published at or after this timestamp |
| `end_time_ms` | integer | Only return events published at or before this timestamp |
| `start` | integer | Ably-compatible alias for `start_time_ms` |
| `end` | integer | Ably-compatible alias for `end_time_ms` |

Response:

```json
{
  "items": [
    {
      "stream_id": "abc123",
      "serial": 42,
      "published_at_ms": 1712678400000,
      "event": "member_added",

      "connection_id": "123.456",
      "dead_node_id": null,
      "payload_size_bytes": 128,
      "presence_event": { /* full event payload */ }
    }
  ],
  "direction": "newest_first",
  "limit": 100,
  "has_more": false,
  "next_cursor": null,

    "start_serial": null,

```

### Inspect Stream State

```
GET /apps/{appId}/channels/{channelName}/presence/history/state
```

Returns the authoritative stream runtime state including durable health, continuity status, and retention statistics.

### Reconstruct Snapshot

```
GET /apps/{appId}/channels/{channelName}/presence/history/snapshot
```

Reconstructs effective membership at a point in time by replaying retained events.

Query parameters:

| Parameter | Type | Description |
|---|---|---|
| `at_time_ms` | integer | Reconstruct membership as of this timestamp |
| `at` | integer | Ably-compatible alias for `at_time_ms` |
| `at_serial` | integer | Reconstruct membership as of this serial number |

If no parameters are given, returns the snapshot from all retained events (latest state).

Response:

```json
{
  "channel": "presence-room",
  "members": [
    {
      "user_id": "user-1",
      "last_event": "member_added",
      "last_event_serial": 42,
      "last_event_at_ms": 1712678400000
    }
  ],
  "member_count": 1,
  "events_replayed": 42,
  "snapshot_serial": 42,
  "snapshot_time_ms": 1712678400000,
  "continuity": { /* same shape as history continuity */ }
}
```

### Reset Stream

```
POST /apps/{appId}/channels/{channelName}/presence/history/reset
```

Purges all retained history and rotates the stream ID. This is an operator action for recovering from unresolvable state.

Request body:

```json
{
  "confirm_channel": "presence-room",
  "confirm_operation": "reset",
  "reason": "Stream corrupted after node failure",
  "requested_by": "operator@example.com"
}
```

## Event Model

Each presence history item represents one state transition:

| Field | Description |
|---|---|
| `event` | `member_added` or `member_removed` |
| `cause` | Why the transition happened |
| `user_id` | The member whose state changed |
| `connection_id` | Which connection triggered the event (nullable) |
| `dead_node_id` | Set when cause is `orphan_cleanup` |
| `serial` | Monotonic per-stream sequence number |
| `published_at_ms` | Wall-clock timestamp of the transition |

### Cause Taxonomy

| Cause | Meaning |
|---|---|
| `join` | Normal user subscription to a presence channel |
| `disconnect` | Normal user disconnection or unsubscription |
| `orphan_cleanup` | Cluster detected a dead node and cleaned up its members |
| `timeout` | Presence membership timed out |
| `forced_disconnect` | Server forcibly terminated the connection |

## Continuity And Degradation

The `continuity` object on every response tells you whether the history stream is healthy:

- `complete: true` — all events within the retained window are present
- `truncated_by_retention: true` — older events have been evicted by retention policy
- `degraded: true` — a write failure has occurred; some events may be missing

When `degraded` is true, the stream enters a fail-closed state. Clients should treat the history as potentially incomplete. Operators can inspect the stream state endpoint and reset the stream if needed.

## Deduplication

Presence history automatically deduplicates:

1. **Exact duplicates** — same `dedupe_key` (e.g., from multi-node fanout) are collapsed
2. **Consecutive user state** — a second `member_added` for the same user without an intervening `member_removed` is suppressed

## Retention

Retention is configurable at three levels:

1. **Time window** — events older than `retention_window_seconds` are evicted
2. **Event count** — optional `max_events_per_channel` cap
3. **Byte budget** — optional `max_bytes_per_channel` cap

Eviction happens on the write path. There is no background compaction process.

## Cursors

Cursors are opaque base64-encoded strings. A cursor is valid only for the same app, channel, direction, and bounds. Using a cursor with different parameters returns `400`.

Do not parse, modify, or store cursors long-term.

## SDK Usage

### Server SDKs

Server SDKs call the REST API directly with HMAC authentication.

::code-group
```typescript [Node.js]
const sockudo = new Sockudo({ host: '127.0.0.1', port: 6001, ... });

// Fetch history
const page = await sockudo.channelPresenceHistory('presence-room', {
  limit: 50,
  direction: 'newest_first',
});

// Paginate
while (page.has_more && page.next_cursor) {
  page = await sockudo.channelPresenceHistory('presence-room', {
    cursor: page.next_cursor,
  });
}

// Snapshot
const snapshot = await sockudo.channelPresenceSnapshot('presence-room', {
  at_time_ms: Date.now() - 3600000, // 1 hour ago
});
```

```rust [Rust]
let page = sockudo
    .channel_presence_history("presence-room", None)
    .await?;

let snapshot = sockudo
    .channel_presence_snapshot("presence-room", Some(&PresenceSnapshotParams {
        at_time_ms: Some(now_ms - 3_600_000),
        ..Default::default()
    }))
    .await?;
```

```go [Go]
page, err := client.ChannelPresenceHistory("presence-room", PresenceHistoryParams{
    Limit: intPtr(50),
})

snapshot, err := client.ChannelPresenceSnapshot("presence-room", PresenceSnapshotParams{
    AtTimeMS: int64Ptr(time.Now().UnixMilli() - 3600000),
})
```

```php [PHP]
$page = $sockudo->getPresenceHistory('presence-room', [
    'limit' => 50,
    'direction' => 'newest_first',
]);

$snapshot = $sockudo->getPresenceSnapshot('presence-room', [
    'at_time_ms' => (int)(microtime(true) * 1000) - 3600000,
]);
```
::

### Client SDKs (via Backend Proxy)

Client SDKs do not hold the HMAC secret. They call a backend proxy endpoint that you provide:

```typescript
const sockudo = new Sockudo({
  // ...
  presenceHistory: {
    endpoint: '/api/presence-history',
  },
});

const channel = sockudo.subscribe('presence-room');
const page = await channel.history({ limit: 50 });

while (page.hasNext()) {
  const nextPage = await page.next();
}

const snapshot = await channel.snapshot({
  at_time_ms: Date.now() - 3600000,
});
```

Your backend proxy receives `POST /api/presence-history` with:
```json
{
  "channel": "presence-room",
  "params": { "limit": 50 },
  "action": "history"
}
```

The proxy must verify the user's session, then call the Sockudo REST API using a server SDK with HMAC auth.

## How This Differs From Using A Database

Presence history is not a general-purpose database. Key differences:

- **Retention-bounded** — old events are automatically evicted. This is a sliding window, not permanent storage.
- **Single-channel scope** — no cross-channel queries. Each channel is an independent stream.
- **No arbitrary queries** — you can filter by serial range or time range, but not by user, cause, or other fields.
- **Monotonic serials** — events are ordered by an internal serial number, not by wall-clock time alone.
- **Best-effort durability** — the memory backend loses data on restart. Use presence history for operational insight, not as a system of record.

If you need permanent presence audit logs, replicate presence history to an external database using webhooks or a queue consumer.

## Errors

| HTTP Status | Code | Meaning |
|---|---|---|
| 400 | `invalid_input` | Invalid direction, bounds, or cursor |
| 422 | `feature_disabled` | Presence history disabled by app policy |
| 400 | `invalid_input` | Channel is not a presence channel |
