## Overview

Echo control determines whether a WebSocket connection receives events that it published itself. By default, Sockudo delivers all messages to all subscribers on a channel, including the sender. Echo control lets V2 connections suppress their own messages at two levels: connection-wide or per-message.

::callout{type="info"}
Echo control is a **protocol V2 only** feature. V1 (Pusher-compatible) connections always receive their own messages, matching standard Pusher behavior.
::

## Configuration

Echo control can be configured globally:

```toml [config/config.toml]
[echo_control]
enabled = true
default_echo_messages = true
```

Or via environment variables:

```bash
ECHO_CONTROL_ENABLED=true
ECHO_CONTROL_DEFAULT_ECHO_MESSAGES=true
```

When `enabled = false`, both the `echo_messages` query parameter and `extras.echo` are ignored.

## Two Levels of Control

### Connection-Level

Disable echo for all messages on a connection by setting the `echo_messages` query parameter when connecting:

```
wss://your-server.com/app/your-key?protocol=2&echo_messages=false
```

With `echo_messages=false`, the connection will not receive any events that it triggered or that were published with its `socket_id` excluded. If the parameter is omitted, the server uses `default_echo_messages`.

### Per-Message Level

Override the connection setting for individual messages using the `extras.echo` field:

```json
POST /apps/{app_id}/events
{
  "name": "collaborative-edit",
  "channel": "document-123",
  "data": "{\"op\":\"insert\",\"text\":\"hello\"}",
  "socket_id": "1234.5678",
  "extras": {
    "echo": true
  }
}
```

Even if socket `1234.5678` connected with `echo_messages=false`, it will receive this specific message because `extras.echo` is `true`.

## Precedence Rules

| Connection `echo_messages` | Message `extras.echo` | Sender Receives? |
|---|---|---|
| `true` (default) | not set | Yes |
| `true` | `false` | No |
| `false` | not set | No |
| `false` | `true` | Yes |

The rule is straightforward: **per-message `extras.echo` always wins**. When `extras.echo` is not set, the connection-level setting applies.

## V1 Behavior

V1 connections do not support echo control:

- The `echo_messages` query parameter is ignored on V1 connections
- The `extras.echo` field is stripped before delivery to V1 connections
- V1 connections always receive their own messages (standard Pusher behavior)

To exclude a V1 connection from receiving its own event, use the existing Pusher `socket_id` exclusion mechanism in the REST API:

```json
POST /apps/{app_id}/events
{
  "name": "my-event",
  "channel": "my-channel",
  "data": "{}",
  "socket_id": "1234.5678"
}
```

This excludes the specified socket from delivery on both V1 and V2 connections.

## REST API Publishing

When publishing via the REST API without a `socket_id`, all subscribers receive the message regardless of echo settings (there is no "sender" to suppress):

```json
POST /apps/{app_id}/events
{
  "name": "server-announcement",
  "channel": "general",
  "data": "{\"message\":\"Server restarting in 5 minutes\"}"
}
```

Echo control only applies when a `socket_id` is specified in the publish request or when a client triggers a client event.

## Client Events

For client events triggered over WebSocket, the sender's socket ID is automatically known. Echo control applies directly:

```javascript
// Connection created with echo_messages=false
const socket = new Sockudo('your-key', {
  protocol: 2,
  echoMessages: false
});

const channel = socket.subscribe('chat-room');

// This client will NOT receive its own client-typing event
channel.trigger('client-typing', { user: 'alice' });

// Override: this specific event WILL echo back
channel.trigger('client-important', { data: 'test' }, {
  extras: { echo: true }
});
```

## Monitoring

Sockudo exposes a Prometheus counter for suppressed echo messages:

```
sockudo_echo_suppressed_total
```

This increments each time a message is suppressed due to echo control. Use it to verify echo control is active:

```promql
rate(sockudo_echo_suppressed_total[5m])
```
