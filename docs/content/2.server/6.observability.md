## Health Endpoints

- `GET /up`: global service health.
- `GET /up/{appId}`: app-specific health.

Behavior:

- `200 OK` with `X-Health-Check: OK` or `DEGRADED`
- `503 Service Unavailable` with `X-Health-Check: ERROR`
- `404 Not Found` for unknown app ID on app-specific checks

## Metrics Endpoint

Prometheus metrics are served on metrics listener (default `:9601`):

```bash
curl http://127.0.0.1:9601/metrics
```

Configure:

```bash
METRICS_ENABLED=true
METRICS_PORT=9601
METRICS_PROMETHEUS_PREFIX=sockudo_
```

## Mutable Message Metrics

Release 4.3 adds Prometheus counters for versioned-message runtime behavior:

- `versioned_message_mutations_total{action,result}`
- `versioned_message_retrieval_total{surface,result}`
- `versioned_history_substitution_total{result}`

Interpretation:

- `action` covers `message.create`, `message.update`, `message.delete`, and `message.append`
- `surface` currently distinguishes latest-visible lookup versus version-history lookup
- `result` distinguishes normal success paths from anomalies such as `auth_failed`, `not_found`, or `missing_latest`

These metrics make it possible to answer:

- are mutable-message mutations being exercised in production?
- are mutation failures operational (auth/not-found) or storage-related?
- is latest-visible history substitution happening normally?
- are retained rows pointing at missing latest winners?

## Annotation Metrics

Release 4.4 adds Prometheus metrics for annotation volume, summary delivery, clipping, and projection rebuilds:

- `annotations_published_total{channel,type}`
- `annotations_deleted_total{channel,type}`
- `annotation_summary_deliveries_total{channel}`
- `annotation_summary_clipped_total{channel,type}`
- `annotation_projection_rebuild_total{channel}`
- `annotation_projection_rebuild_duration_seconds{channel}`

With the default Prometheus prefix these appear as `sockudo_annotations_published_total`, `sockudo_annotation_summary_clipped_total`, and so on.

Use these metrics to answer:

- which channels and annotation types are generating annotation churn?
- are subscribers receiving summary fanout after annotation changes?
- are contributor lists clipping because summaries exceed payload limits?
- are projection caches cold or stale often enough to rebuild on hot channels?

## Usage And Statistics Endpoints

- `GET /usage` returns a memory snapshot from the server process context.
- `GET /stats` returns memory, per-app connection totals, authenticated connection totals, global occupancy totals, presence occupancy totals, and connection metadata attachment counts.

Both endpoints can be disabled via `http_api.usage_enabled: false` or `HTTP_API_USAGE_ENABLED=false`.

## Logging

Control output format and verbosity with:

```bash
DEBUG=true
RUST_LOG=info,sockudo=debug
LOG_OUTPUT_FORMAT=human   # or json
LOG_COLORS_ENABLED=true
LOG_INCLUDE_TARGET=true
```

High-signal release-4.3 logs:

- `Applied versioned message.create`
- `Applied versioned message.update`
- `Applied versioned message.delete`
- `Applied versioned message.append`
- `Denied versioned message mutation authorization`
- `History row referenced a versioned message without a latest winner; returning stored payload`

High-signal release-4.4 annotation logs:

- `annotation summary clipped`
- `annotation projection rebuild triggered on hot channel`
- `annotation summary failed to fan out through cluster broadcast`
- `annotation event failed to fan out through cluster broadcast`

## Disconnect Cleanup Queue

Async cleanup avoids mass-disconnect stalls:

```bash
CLEANUP_ASYNC_ENABLED=true
CLEANUP_QUEUE_BUFFER_SIZE=50000
CLEANUP_BATCH_SIZE=25
CLEANUP_BATCH_TIMEOUT_MS=50
CLEANUP_WORKER_THREADS=auto
```

`auto` worker mode scales by CPU count.
