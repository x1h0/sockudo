# Push Capacity Planning

This guide turns the binding push capacity model into operator-facing planning
rules. It describes Sockudo dispatch capacity, not guaranteed mobile-device
receipt. Provider quotas, token validity, payload size, regional capacity, and
retry-after behavior remain external limits.

## Deployment Modes

| Mode | Shape | Fit |
| --- | --- | --- |
| Monolith | API, planner, scheduler, workers, store, and queue in one process or small node group. | Development and small production. Use durable store/queue for production. |
| Distributed | API nodes, planner workers, shard workers, provider workers, feedback processors, durable store, and durable broker scaled independently. | Regional, cloud-native, and hyperscale deployments. |

Async publish returns `202` only after the initial status record is persisted
and the publish-log event is durable. Synchronous provider fanout from the
Publish API is not a supported capacity-planning mode.

## Backend Tier Selection

| Storage backend | Tier | Use when | Do not claim |
| --- | --- | --- | --- |
| memory | tests/dev only | Unit tests and local demos. | Production durability or multi-node registry. |
| PostgreSQL | small production | You need SQL correctness, migrations, and regional scale under about 10M devices unless benchmarked. | 100M channel fanout or 1B registry without partitioning proof. |
| MySQL | small production | MySQL is already the operational database and push schema remains simple. | Hyperscale fanout/status analytics. |
| SurrealDB | small supported tier | Small deployments already using SurrealDB. | Hyperscale without load tests. |
| DynamoDB | cloud-native production | AWS-native registry, bucketed GSIs, TTL, and capacity alarms. | Low-cost hot-channel fanout without partition design. |
| ScyllaDB | hyperscale recommended | Largest self-managed registries and massive fanout indexes. | Unbounded partitions, tombstone-heavy cleanup, or casual schema changes. |

## Broker Selection

Every supported queue backend must model publish-log, planning, shard, delivery,
result, retry, and DLQ semantics. Broker delay features may optimize scheduled
delivery, but storage is the source of truth for schedules and cancellation.

| Queue backend | Tier | Planning guidance |
| --- | --- | --- |
| memory | tests/dev only | Non-durable, single process. |
| Redis | small production | Prefer Redis Streams. Existing list pop-before-process semantics are not sufficient for reliability mode. |
| Redis Cluster | small/regional production | Use hash-tagged streams and avoid hot slots. Same reliability caveat as Redis lists. |
| NATS JetStream | high-scale candidate | Configure stream retention, max deliver, DLQ subjects, and per-stage subjects. |
| Kafka/Redpanda | primary high-scale candidate | Configure partition counts and keys. One partition is invalid for push scale claims. |
| Pulsar | primary high-scale candidate | Use partitioned topics and shared/key-shared subscriptions. |
| Iggy | high-scale candidate | Supported candidate; hyperscale claims require benchmark evidence. |
| RabbitMQ | regional production | Use durable queues, acks, DLQ, and retention limits. Not the default 100M fanout path. |
| Google Pub/Sub | cloud-native supported path | Strong on GCP with ack deadlines, ordering keys, quotas, and cost planning. |
| SQS | cloud-native supported path | Good for sharded dispatch. Massive planning requires aggressive queue sharding. |
| SNS | publish fanout only | Not a complete worker queue by itself; pair with SQS or another consumer queue. |

## Sizing Formulas

Use these formulas as planning inputs, then validate them with load tests:

| Component | Formula |
| --- | --- |
| API acceptance RPS | `min(configured acceptance_rps, durable status writes/s, publish-log appends/s)` |
| Planner workers | `ceil(publish_log_lag_target / per_worker_plans_per_sec)` with tenant fairness enabled. |
| Shard jobs | `ceil(expected_recipients / PUSH_FANOUT_SHARD_SIZE)` when fanout is at or above the fast threshold. |
| Storage token pages | `ceil(expected_recipients / PUSH_FANOUT_PAGE_SIZE)` |
| Provider batches | `ceil(expected_recipients / PUSH_PROVIDER_BATCH_SIZE)` before provider-specific splitting. |
| Provider workers | `ceil(target_provider_attempts_per_sec * provider_latency_sec / batch_size)` capped by provider quota and tenant in-flight caps. |
| Broker partitions | At least enough partitions to spread `(app_id, provider, publish_id)` lanes across worker concurrency without hot partitions. |
| Store write capacity | Status, shard checkpoints, delivery attempts, feedback events, cleanup updates, and analytics rollups at peak retry rate. |

Worked example for a 100M channel with `PUSH_FANOUT_SHARD_SIZE=100000`,
`PUSH_FANOUT_PAGE_SIZE=1000`, and `PUSH_PROVIDER_BATCH_SIZE=500`:

| Metric | Result |
| --- | ---: |
| Shard jobs | `1,000` |
| Storage pages | `100,000` |
| Delivery batches | `200,000` |
| Delivery result events | `200,000+` |

At 20,000 provider-token attempts per second after quota smoothing, the dispatch
envelope is about 83 minutes before retries. At 100,000 attempts per second, it
is about 17 minutes. These are Sockudo dispatch envelopes only.

## Provider Quota Uplift

Plan quota uplift before increasing worker counts:

- FCM: respect `Retry-After`, back off 429/5xx with jitter, and treat
  `UNREGISTERED` as cleanup signal.
- APNs: manage HTTP/2 pools, GOAWAY reconnect, 410 cleanup, 429/5xx backoff, and
  topic/key isolation.
- Web Push: smooth by push-service origin, handle 404/410 cleanup, 429/5xx
  backoff, and VAPID auth failures.
- HMS/WNS: benchmark with provider-specific quotas, token refresh, invalid-token
  cleanup, and throttling headers.

More Sockudo workers must not override a provider circuit breaker, storage
throttle, broker quota, or tenant cap.

## Benchmark Harness

Sockudo includes two push benchmark surfaces:

| Harness | Command | Measures |
| --- | --- | --- |
| Internal Criterion | `make push-benchmark-internal` | Admission, fast-path planning, shard creation, shard expansion, queue produce/consume/ack, provider payload rendering, and mock provider dispatch. |
| Live HTTP load | `make push-benchmark ARGS="--mode all --devices 10000 --channels 100 --concurrency 50 --duration 120"` | REST publish admission, registry seeding, publish latency percentiles, sampled status counters, and hot-channel fanout against a running Sockudo node. |
| Scenario suite | `make push-benchmark-scenarios` | Dry-run benchmark plan for admission, registry fanout, hot channels, provider-shaped fake-token runs, quota sweeps, and soak tests. |
| Mock provider | `make push-mock-provider ARGS="--profile throttle"` | Provider response classification, retry-after behavior, throttling, auth failures, invalid-token cleanup, and outage simulation without calling external providers. |
| Real canary | `make push-provider-canary ARGS="--provider fcm --max-sends 5 --execute ..."` | Small bounded real-provider verification only; not a scale benchmark. |

Suggested runs:

```bash
# API admission only against an already-seeded registry.
make push-benchmark ARGS="--mode admission --channels 100 --concurrency 100 --duration 120"

# Seed and test many small channels.
make push-benchmark ARGS="--mode all --devices 100000 --channels 100 --concurrency 100 --duration 300"

# Hot-channel fanout pressure.
make push-benchmark ARGS="--mode all --devices 100000 --channels 1 --hot-channel --concurrency 50 --duration 120"

# Internal pipeline baselines.
make push-benchmark-internal
```

Run with a mock or disabled provider path first to establish Sockudo capacity.
Then repeat with real provider credentials to measure provider quota, retry,
and network limits separately.

## Cost Caveats

Cost depends on raw fanout volume, retry rate, feedback granularity, status
retention, analytics retention, hot-channel sharding, and provider quota
smoothing. DynamoDB GSIs, Pub/Sub messages, SQS requests, Kafka retention,
ScyllaDB compaction, and SQL index maintenance can dominate cost at different
tiers. Status and analytics retention should be bounded before load tests.

## Failure Runbook

| Symptom | First checks | Operator action |
| --- | --- | --- |
| Publish API returns `429` | Tenant acceptance RPS, fanout caps, daily quotas. | Raise tenant quota only after confirming provider and broker capacity. |
| Publish API returns `503` | Status store writes, publish-log append latency, queue lag. | Stop accepting unbounded work; scale or repair the durable boundary first. |
| Planner lag grows | Publish-log lag, shard backlog, store page latency. | Add planner/shard workers only if store and broker are healthy. |
| Provider throttling | Provider 429/5xx, retry-after, circuit breaker state. | Reduce concurrency, request provider quota uplift, keep tenant fairness enabled. |
| Feedback lag grows | Result queue depth, status write latency, malformed feedback DLQ. | Scale feedback processors or repair status store before raising dispatch. |
| Stale token spike | Provider invalid-token classes, cleanup job lag. | Verify provider credentials and run bounded stale cleanup batches. |
| DLQ grows | Retry exhaustion, malformed jobs, credential failures. | Replay only after fixing root cause; keep payload and credentials redacted. |

Completion evidence for any public scale claim must include load, chaos, and soak
tests for the chosen storage and broker combination.
