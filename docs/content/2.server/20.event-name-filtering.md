## Overview

Event name filtering allows V2 clients to specify which event names they want to receive when subscribing to a channel. The server evaluates the filter and only delivers matching events, reducing bandwidth and client-side processing.

::callout{type="info"}
Event name filtering is a **protocol V2 only** feature. V1 (Pusher-compatible) connections receive all events on subscribed channels without any server-side filtering.
::

## Configuration

Event name filtering can be configured globally:

```toml [config/config.toml]
[event_name_filtering]
enabled = true
max_events_per_filter = 50
max_event_name_length = 200
```

Or via environment variables:

```bash
EVENT_NAME_FILTERING_ENABLED=true
EVENT_NAME_FILTERING_MAX_EVENTS_PER_FILTER=50
EVENT_NAME_FILTERING_MAX_EVENT_NAME_LENGTH=200
```

When `enabled = false`, `filter.events` is ignored and subscriptions receive all event names.

## Subscribing with an Event Filter

Include a `filter` object with an `events` array in the subscription data:

```json
{
  "event": "sockudo:subscribe",
  "data": {
    "channel": "market-data",
    "filter": {
      "events": ["price-update", "trade-executed"]
    }
  }
}
```

This subscription only receives `price-update` and `trade-executed` events on the `market-data` channel. All other events published to the channel are silently dropped for this subscriber.

### Client SDK Example

```javascript
const channel = socket.subscribe('market-data', {
  filter: {
    events: ['price-update', 'trade-executed']
  }
});

// Only fires for price-update and trade-executed events
channel.bind('price-update', (data) => {
  console.log('Price:', data);
});
```

## Compound Filters

Event name filters can be combined with [tag filters](/server/tag-filtering) in a single subscription. Both conditions must match for a message to be delivered:

```json
{
  "event": "sockudo:subscribe",
  "data": {
    "channel": "market-data",
    "filter": {
      "events": ["price-update"],
      "tags": "symbol == 'AAPL' OR symbol == 'GOOGL'"
    }
  }
}
```

This subscriber only receives `price-update` events where the tag `symbol` is `AAPL` or `GOOGL`. A `trade-executed` event with `symbol=AAPL` would not match because the event name is not in the filter.

### Filter Evaluation

When both `events` and `tags` are present:

1. Check if the event name is in the `events` list
2. If yes, evaluate the tag filter expression
3. Deliver only if **both** match

If only `events` is set, tag filtering is skipped. If only `tags` is set, all event names are accepted and only tags are evaluated.

## Validation Rules

| Constraint | Limit |
|---|---|
| Maximum event names per filter | `event_name_filtering.max_events_per_filter` (default `50`) |
| Maximum length per event name | `event_name_filtering.max_event_name_length` (default `200`) |
| Empty events array | Treated as no filter (all events delivered) |

The server rejects subscriptions that exceed these limits with an error response:

```json
{
  "event": "sockudo:error",
  "data": {
    "code": 4301,
    "message": "Event filter exceeds maximum of 50 event names"
  }
}
```

## V1 Behavior

V1 connections do not support event name filtering:

- The `filter.events` field in subscription data is ignored
- All events published to a channel are delivered to all V1 subscribers
- No error is returned; the filter is silently ignored to maintain compatibility

## Use Cases

**Chat application** -- subscribe only to message events, ignore typing indicators:

```json
{
  "filter": {
    "events": ["message-sent", "message-edited", "message-deleted"]
  }
}
```

**Dashboard** -- subscribe only to the metrics you display:

```json
{
  "filter": {
    "events": ["cpu-usage", "memory-usage", "disk-io"]
  }
}
```

**Notification panel** -- receive only high-priority notification types:

```json
{
  "filter": {
    "events": ["alert-critical", "alert-warning"],
    "tags": "service == 'payments'"
  }
}
```

## Monitoring

Sockudo exposes a Prometheus counter for events suppressed by event name filters:

```
sockudo_event_filter_suppressed_total
```

This increments each time a message is not delivered to a subscriber because its event name did not match the subscription filter:

```promql
rate(sockudo_event_filter_suppressed_total[5m])
```

A high suppression rate indicates effective filtering -- clients are only receiving the events they need.
