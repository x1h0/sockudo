# Redis Sentinel + TLS test fixture

An **opt-in** local fixture for exercising the Redis adapter's Sentinel path with
TLS and mutual TLS (client certificates). It is not part of the default test run
or CI: the compose services live behind the `sentinel-tls` profile, and the
integration tests are `#[ignore]`d (matching `sockudo-rate-limiter/tests/redis_limiter_live.rs`).

> Commands below use `docker-compose` (v1, hyphen) to match the repo's Makefile.
> If you only have the Compose v2 plugin, substitute `docker compose`.

## What it provides

- `redis-master-tls` — a TLS-only Redis master (`tls-auth-clients yes`, password `masterpass`).
- `redis-sentinel-tls-1..3` — three TLS Sentinels monitoring `mymaster` (quorum 2).
- A throwaway CA plus server and client certificates generated by
  [`generate-certs.sh`](./generate-certs.sh) into `certs/` (gitignored).

## Quick start

```bash
# 1. Generate certs and start the fixture (also prints the /etc/hosts hint).
make sentinel-tls-up

# 2. Run the gated live tests against it.
make sentinel-tls-test

# 3. Tear down.
make sentinel-tls-down
```

`make sentinel-tls-test` is equivalent to:

```bash
SOCKUDO_SENTINEL_TLS=1 SOCKUDO_MASTER_TLS=1 \
  SOCKUDO_REDIS_PASSWORD=masterpass \
  SOCKUDO_TLS_CA_PATH=tests/sentinel-tls/certs/ca.crt \
  SOCKUDO_TLS_CLIENT_CERT_PATH=tests/sentinel-tls/certs/client.crt \
  SOCKUDO_TLS_CLIENT_KEY_PATH=tests/sentinel-tls/certs/client.key \
  cargo test -p sockudo-adapter --features redis \
    --test redis_sentinel_live -- --ignored --nocapture
```

## Host networking note

Sentinel hands clients the *master's* address. The fixture monitors the master at
`host.docker.internal:16380` so that address is reachable both from inside the
Sentinel containers (via the host gateway) and from a host-run test. On most
platforms `host.docker.internal` is only resolvable inside containers, so for the
host-run test add this to `/etc/hosts`:

```
127.0.0.1 host.docker.internal
```

Note that `SOCKUDO_TLS_INSECURE=1` only relaxes certificate/hostname verification;
it does **not** fix name resolution, so the `/etc/hosts` entry is the reliable
option for fully local runs.

## Pointing the tests at a real deployment

The same env knobs work against any reachable Sentinel deployment (e.g. a staging
cluster), which is the most faithful validation of production credentials and
certificates:

| Variable | Meaning | Default |
| --- | --- | --- |
| `SOCKUDO_SENTINEL_HOSTS` | Comma-separated `host:port` sentinels | `127.0.0.1:26379,127.0.0.1:26380,127.0.0.1:26381` |
| `SOCKUDO_SENTINEL_MASTER_NAME` | Monitored master name | `mymaster` |
| `SOCKUDO_REDIS_USERNAME` / `SOCKUDO_REDIS_PASSWORD` | Master/replica auth | unset |
| `SOCKUDO_SENTINEL_USERNAME` / `SOCKUDO_SENTINEL_PASSWORD` | Sentinel auth | unset |
| `SOCKUDO_SENTINEL_TLS` | Enable TLS to the sentinels | off |
| `SOCKUDO_MASTER_TLS` | Enable TLS to the master/replica | off |
| `SOCKUDO_TLS_CA_PATH` | PEM CA path (private CAs) | system roots |
| `SOCKUDO_TLS_CLIENT_CERT_PATH` / `SOCKUDO_TLS_CLIENT_KEY_PATH` | mTLS client cert + key | unset |
| `SOCKUDO_TLS_INSECURE` | Skip cert/hostname verification (dangerous) | off |
| `SOCKUDO_REDIS_DB` | Database index | `0` |

> The certificates generated here are throwaway material for local testing only.
> Never commit `certs/` or reuse it elsewhere.
